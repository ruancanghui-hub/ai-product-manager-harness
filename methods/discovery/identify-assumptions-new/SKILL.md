# Identify Assumptions - New Product (新产品假设识别)

## 目的

针对新产品或新业务线的功能想法，系统化识别全方位的风险假设，避免基于未经验证的假设投入大量资源，确保决策基于证据而非猜测。

## 核心概念

```
新产品假设识别八象限
  │
  ├─ 价值假设 (Value)
  │   └─ 用户真的有这个问题吗？
  │
  ├─ 可用性假设 (Usability)
  │   └─ 用户能理解并使用产品吗？
  │
  ├─ 可行性假设 (Feasibility)
  │   └─ 我们能构建出来吗？
  │
  ├─ 商业可行性假设 (Viability)
  │   └─ 能持续盈利吗？
  │
  ├─ 市场假设 (Market)
  │   └─ 市场规模足够大吗？
  │
  ├─ 战略假设 (Strategy)
  │   └─ 与公司战略一致吗？
  │
  ├─ 团队假设 (Team)
  │   └─ 团队有能力实现吗？
  │
  └─ 资源假设 (Resource)
      └─ 有足够资源支撑吗？
```

## 何时使用

- ✅ 新产品从 0 到 1 的初始验证
- ✅ 业务线扩展，需要评估可行性
- ✅ 寻求投资前，识别关键风险
- ✅ 制定产品路线图前
- ✅ 团队对产品方向有分歧

## 使用步骤

### Step 1: 准备阶段 - 明确产品想法

**输入材料**:
1. **产品概念**: 产品是什么？解决什么问题？
2. **目标用户**: 为谁设计？
3. **价值主张**: 为什么用户会选择我们？
4. **商业模式**: 如何盈利？

**准备问题**:
```
- 我们要构建什么产品？
- 这个产品想解决什么问题？
- 我们为什么认为这是一个好机会？
- 如果我们错了，最可能错在哪里？
```

**输出格式**:
```
## 产品想法明确

- 产品名称：[名称]
- 核心价值：[一句话描述]
- 目标用户：[用户画像]
- 商业模式：[盈利方式]
```

### Step 2: 价值假设识别 (20-30 分钟)

**核心问题**: 用户真的有这个问题吗？

**识别方法**:

#### 2.1 问题假设
```
问题：我们认为用户有这个问题，但真的有吗？

示例：
产品：宠物健康追踪 App
假设：宠物主人担心宠物健康，愿意追踪
验证方式：
- 用户访谈：询问宠物主人的担忧
- 数据调研：宠物医疗市场规模
- 竞品分析：现有宠物健康产品
```

#### 2.2 解决方案假设
```
问题：我们的解决方案是最佳方案吗？

示例：
产品：宠物健康追踪 App
假设：App 是宠物主人追踪健康的最佳方式
验证方式：
- 用户访谈：用户现在如何追踪？
- 竞品分析：其他解决方案的优劣
```

#### 2.3 支付意愿假设
```
问题：用户愿意为此付费吗？

示例：
产品：宠物健康追踪 App
假设：宠物主人愿意为健康追踪功能付费
验证方式：
- 价格敏感度测试
- 竞品付费模式调研
- 预售或众筹验证
```

**输出格式**:
```
## 价值假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| VAL-01 | 用户有 X 问题 | 高/中/低 | ... | ... |
| VAL-02 | 用户愿意为 X 付费 | 高/中/低 | ... | ... |
```

### Step 3: 可用性假设识别 (20-30 分钟)

**核心问题**: 用户能理解并使用产品吗？

**识别方法**:

#### 3.1 理解性假设
```
问题：用户能理解产品的价值吗？

示例：
产品：宠物健康追踪 App
假设：用户能快速理解 App 的核心价值
验证方式：
- 落地页测试：价值主张转化率
- 用户访谈：5 秒测试
```

#### 3.2 操作性假设
```
问题：用户能顺利完成关键任务吗？

示例：
产品：宠物健康追踪 App
假设：用户能独立完成宠物信息录入
验证方式：
- 原型测试：观察用户操作流程
- 可用性测试：任务完成率和时间
```

**输出格式**:
```
## 可用性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| USA-01 | 用户能理解 X 价值 | 高/中/低 | ... | ... |
| USA-02 | 用户能完成 Y 任务 | 高/中/低 | ... | ... |
```

### Step 4: 可行性假设识别 (20-30 分钟)

**核心问题**: 我们能构建出来吗？

**识别方法**:

#### 4.1 技术可行性
```
问题：技术上能实现吗？

示例：
产品：宠物健康追踪 App
假设：我们能实现准确的宠物健康数据分析
验证方式：
- 技术调研：现有技术方案
- 技术预研：原型验证
```

#### 4.2 资源可行性
```
问题：有足够的资源构建吗？

示例：
产品：宠物健康追踪 App
假设：团队能在 3 个月内完成 MVP
验证方式：
- 资源盘点：团队可用资源
- 工期估算：详细开发计划
```

**输出格式**:
```
## 可行性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| FEA-01 | 技术上能实现 X | 高/中/低 | ... | ... |
| FEA-02 | 有足够资源构建 | 高/中/低 | ... | ... |
```

### Step 5: 商业可行性假设识别 (20-30 分钟)

**核心问题**: 能持续盈利吗？

**识别方法**:

#### 5.1 盈利模式假设
```
问题：盈利模式可行吗？

示例：
产品：宠物健康追踪 App
假设：订阅制能覆盖成本并盈利
验证方式：
- 财务模型：成本收益分析
- 竞品调研：同类产品盈利模式
```

#### 5.2 获客成本假设
```
问题：获客成本可控吗？

示例：
产品：宠物健康追踪 App
假设：能以低于 50 元的成本获取一个用户
验证方式：
- 渠道测试：不同渠道的获客成本
- 竞品调研：行业平均获客成本
```

#### 5.3 用户生命周期价值假设
```
问题：用户生命周期价值足够高吗？

示例：
产品：宠物健康追踪 App
假设：用户平均生命周期价值能达 300 元
验证方式：
- 竞品数据：同类产品 LTV
- 用户调研：支付意愿和频次
```

**输出格式**:
```
## 商业可行性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| VIA-01 | X 盈利模式可行 | 高/中/低 | ... | ... |
| VIA-02 | 获客成本可控 | 高/中/低 | ... | ... |
```

### Step 6: 市场假设识别 (20-30 分钟)

**核心问题**: 市场规模足够大吗？

**识别方法**:

#### 6.1 市场规模假设
```
问题：目标市场足够大吗？

示例：
产品：宠物健康追踪 App
假设：中国宠物健康市场足够支撑规模化
验证方式：
- 市场调研：市场规模数据
- TAM/SAM/SOM 分析
```

#### 6.2 市场增长假设
```
问题：市场在增长吗？

示例：
产品：宠物健康追踪 App
假设：宠物健康市场在未来 5 年持续增长
验证方式：
- 行业报告：市场增长率
- 趋势分析：宠物经济趋势
```

#### 6.3 市场时机假设
```
问题：现在是进入的好时机吗？

示例：
产品：宠物健康追踪 App
假设：当前市场成熟度适合进入
验证方式：
- 时机分析：市场发展阶段
- 竞品动态：竞品进入时间
```

**输出格式**:
```
## 市场假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| MAR-01 | 市场规模足够大 | 高/中/低 | ... | ... |
| MAR-02 | 市场持续增长 | 高/中/低 | ... | ... |
```

### Step 7: 战略假设识别 (20 分钟)

**核心问题**: 与公司战略一致吗？

**识别方法**:

#### 7.1 战略一致性
```
问题：与公司战略方向一致吗？

示例：
产品：宠物健康追踪 App
假设：符合公司「宠物生态」战略方向
验证方式：
- 战略文档：公司战略规划
- 利益相关者访谈
```

#### 7.2 竞争优势
```
问题：我们有竞争优势吗？

示例：
产品：宠物健康追踪 App
假设：我们比竞品有更强的技术优势
验证方式：
- 竞品分析：竞争优势对比
- 能力盘点：团队核心能力
```

**输出格式**:
```
## 战略假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| STR-01 | 与战略方向一致 | 高/中/低 | ... | ... |
| STR-02 | 有竞争优势 | 高/中/低 | ... | ... |
```

### Step 8: 团队假设识别 (20 分钟)

**核心问题**: 团队有能力实现吗？

**识别方法**:

#### 8.1 团队能力
```
问题：团队有实现产品的能力吗？

示例：
产品：宠物健康追踪 App
假设：团队有宠物行业经验和移动开发能力
验证方式：
- 团队盘点：团队能力矩阵
- 招聘计划：能力缺口补充
```

#### 8.2 团队激情
```
问题：团队对这个方向有激情吗？

示例：
产品：宠物健康追踪 App
假设：团队对宠物健康方向有热情
验证方式：
- 团队访谈：了解团队意愿
- 愿景共识：团队对愿景的认同度
```

**输出格式**:
```
## 团队假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| TMA-01 | 团队有能力实现 | 高/中/低 | ... | ... |
| TMA-02 | 团队有激情 | 高/中/低 | ... | ... |
```

### Step 9: 资源假设识别 (20 分钟)

**核心问题**: 有足够资源支撑吗？

**识别方法**:

#### 9.1 资金资源
```
问题：有足够的资金支持吗？

示例：
产品：宠物健康追踪 App
假设：现有资金能支撑到产品盈利
验证方式：
- 财务规划：资金消耗速度
- 融资计划：融资节点和金额
```

#### 9.2 时间资源
```
问题：有足够的时间吗？

示例：
产品：宠物健康追踪 App
假设：能在资金耗尽前验证产品
验证方式：
- 时间规划：里程碑时间表
- 风险评估：潜在延期风险
```

**输出格式**:
```
## 资源假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| RES-01 | 资金足够支撑 | 高/中/低 | ... | ... |
| RES-02 | 时间足够验证 | 高/中/低 | ... | ... |
```

### Step 10: 整合假设清单 (30 分钟)

**整合方法**:
1. 将所有假设汇总到一张表
2. 标注假设间的依赖关系
3. 识别「关键路径假设」

**假设依赖关系图**:
```
产品想法
  │
  ├─ 市场假设 (市场存在)
  │   └─ 价值假设 (用户有需求)
  │       └─ 商业假设 (能盈利)
  │
  ├─ 团队假设 (有能力)
  │   └─ 资源假设 (有资源)
  │       └─ 可行性假设 (能实现)
  │
  └─ 战略假设 (方向对)
```

**关键路径假设识别**:
```
关键路径假设：如果这些假设不成立，产品就会失败

示例：
1. 宠物主人真的担心宠物健康（价值）
2. 宠物主人愿意为健康追踪付费（价值）
3. 我们能以低于 50 元的成本获取用户（商业）
4. 市场规模能支撑规模化（市场）
```

**输出格式**:
```
## 产品假设总表

| 假设ID | 类型 | 描述 | 风险等级 | 验证方式 | 优先级 |
|--------|------|------|---------|---------|--------|
| VAL-01 | 价值 | ... | 高 | ... | P0 |
| MAR-01 | 市场 | ... | 高 | ... | P0 |
| ... | ... | ... | ... | ... | ... |

## 关键路径假设（必须验证）
...

## 高风险假设（优先验证）
...

## 中等风险假设（建议验证）
...

## 低风险假设（可跳过验证）
...
```

## 常见错误

- ❌ 只关注价值假设，忽视市场、团队、资源
- ❌ 假设描述过于笼统，无法验证
- ❌ 过度乐观，低估风险等级
- ❌ 没有识别关键路径假设
- ❌ 忽视假设间的依赖关系

## 与其他方法的关系

- **前置方法**: brainstorm-ideas-new（功能发散）
- **并行方法**: prioritize-assumptions（假设排序）
- **后置方法**: brainstorm-experiments-new（设计验证实验）
