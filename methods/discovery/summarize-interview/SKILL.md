# Summarize Interview (访谈总结)

## 目的

将用户访谈的录音或笔记整理成结构化的洞察文档,提取关键发现、用户原话、行为模式和需求洞察,为产品决策提供可靠依据。

## 核心概念

```
访谈总结结构
  |
  +-- 基本信息
  |    +-- 访谈对象、时间、方式
  |
  +-- 关键发现
  |    +-- 3-5 个最重要的洞察
  |
  +-- 用户原话
  |    +-- 支持发现的真实引用
  |
  +-- 行为模式
  |    +-- 观察到的行为特征
  |
  +-- 需求洞察
  |    +-- 显性需求和隐性需求
  |
  +-- 行动建议
       +-- 基于洞察的产品建议
```

## 何时使用

- 完成用户访谈后,整理访谈内容
- 需要向团队分享访谈洞察
- 多次访谈后,综合分析用户需求
- 基于访谈洞察制定产品决策
- 积累用户研究素材

## 使用步骤

### Step 1: 准备阶段 - 收集访谈材料

**输入材料**:
1. 访谈录音文件
2. 访谈笔记
3. 访谈脚本
4. 访谈目标

**准备工作**:
- 完整听一遍录音
- 整理笔记和关键点
- 标记重要片段时间点
- 记录用户原话

**输出格式**:
```
访谈材料准备

- 访谈对象:[姓名/编号]
- 访谈时间:[日期]
- 访谈方式:[面对面/视频/电话]
- 访谈时长:[时长]
- 访谈人员:[访谈者]
- 录音文件:[文件名]
- 笔记文件:[文件名]
```

### Step 2: 提取关键发现 (20-30 分钟)

**关键发现定义**: 最重要、最有价值的洞察

**提取方法**:

#### 2.1 识别惊讶点
```
问题:有什么是你之前不知道或感到惊讶的?

示例:
- 用户使用场景与预期不同
- 用户最痛点不是我们以为的
- 用户决策逻辑出乎意料
```

#### 2.2 识别重复模式
```
问题:有什么在多个用户身上反复出现?

示例:
- 多个用户提到同样痛点
- 多个用户使用类似解决方法
- 多个用户有相同决策因素
```

#### 2.3 识别行为矛盾
```
问题:用户说的和做的一致吗?

示例:
- 用户说重视健康但不运动
- 用户说愿意付费但用免费方案
- 用户说功能重要但不使用
```

#### 2.4 识别未满足需求
```
问题:用户有什么需求没有得到满足?

示例:
- 用户抱怨现有方案不够好
- 用户希望能有某个功能
- 用户自己想出临时解决方案
```

**输出格式**:
```
关键发现

发现 1:[标题]
- 发现描述:[具体描述]
- 重要程度:高/中/低
- 支持证据:[用户原话或行为]
- 产品启示:[对产品的指导意义]

发现 2:[标题]
...

(提取 3-5 个最重要发现)
```

### Step 3: 整理用户原话 (15-20 分钟)

**原话选择标准**:
- 能支持关键发现
- 表达生动、具体
- 包含情感色彩
- 揭示深层需求

**原话整理方法**:

#### 3.1 按主题分类
```
分类维度:
- 痛点相关
- 需求相关
- 行为相关
- 决策相关
- 情感相关

示例:
痛点类:
- "每次都要花半小时找文件,太浪费时间了"
- "系统经常崩溃,我都不知道该怎么办"

需求类:
- "我希望能一键找到所有相关文件"
- "如果能自动保存就好了"

行为类:
- "我通常会先在桌面上建一个临时文件夹"
- "我习惯每半小时手动保存一次"

决策类:
- "我选这个软件主要是因为便宜"
- "看到同事用,我就也试试"

情感类:
- "每次找不到文件都特别焦虑"
- "系统崩溃时真的很崩溃"
```

#### 3.2 记录原话上下文
```
格式:
原话 + 上下文

示例:
原话:"每次都要花半小时找文件,太浪费时间了"
上下文:用户在描述每周整理项目文件时的困扰
场景:周五下班前整理本周工作文件
```

**输出格式**:
```
用户原话记录

| 原话 | 主题 | 上下文 | 支持发现 |
|------|------|--------|---------|
| ... | 痛点 | ... | 发现 1 |
| ... | 需求 | ... | 发现 2 |
```

### Step 4: 分析行为模式 (15-20 分钟)

**行为模式定义**: 用户反复出现的行为特征

**分析方法**:

#### 4.1 现有解决方案
```
问题:用户现在如何解决问题?

分析:
- 用户使用的工具/方法
- 使用频率和场景
- 效果和满意度

示例:
现有方案:用 Excel 手动记录项目进度
频率:每周一次
效果:容易出错、难追踪
满意度:低,觉得麻烦
```

#### 4.2 替代方案
```
问题:用户尝试过哪些替代方案?

分析:
- 尝试过的其他方案
- 为什么放弃
- 还会考虑什么

示例:
尝试过的方案:
- Trello:功能太简单,不适合复杂项目
- JIRA:学习成本太高,团队不接受
- Excel:虽麻烦但熟悉,目前还在用
```

#### 4.3 工作流程
```
问题:用户的完整工作流程是什么?

分析:
- 任务触发点
- 执行步骤
- 完成标准
- 难点环节

示例:
工作流程:
1. 收到项目任务(触发)
2. 创建 Excel 表格(开始)
3. 分配任务给团队(执行)
4. 每周手动更新进度(追踪)
5. 汇报给上级(完成)

难点:步骤 4,手动更新耗时易错
```

#### 4.4 决策触发点
```
问题:什么触发用户做出决策?

分析:
- 触发事件
- 决策时机
- 决策因素
- 决策障碍

示例:
触发事件:项目延期,团队协作混乱
决策时机:项目复盘后,决定改进
决策因素:易用性、价格、团队接受度
决策障碍:学习成本、迁移成本
```

**输出格式**:
```
行为模式分析

现有解决方案:
- 方案:[方法]
- 频率:[使用频率]
- 效果:[效果评估]
- 满意度:[满意程度]

替代方案:
- 尝试过的:[方案列表]
- 放弃原因:[原因]
- 还会考虑:[方案]

工作流程:
- 触发点:[触发事件]
- 步骤:[执行步骤]
- 难点:[困难环节]

决策模式:
- 触发事件:[事件]
- 决策因素:[因素]
- 决策障碍:[障碍]
```

### Step 5: 提炼需求洞察 (15-20 分钟)

**需求分类**:

#### 5.1 显性需求
```
定义:用户明确表达的需求

识别方法:
- 用户直接说"我需要..."
- 用户抱怨缺少某个功能
- 用户提到希望能有...

示例:
显性需求:
- "我需要能自动保存的功能"
- "希望能一键找到所有文件"
- "想要更简单的操作界面"
```

#### 5.2 隐性需求
```
定义:用户未明确表达但实际存在的需求

识别方法:
- 从痛点推断需求
- 从行为推断需求
- 从情感推断需求

示例:
隐性需求:
痛点:"找文件浪费时间" -> 需要智能搜索
行为:"手动保存怕丢失" -> 需要自动备份
情感:"找不到文件很焦虑" -> 需要可靠的组织方式
```

#### 5.3 JTBD 需求
```
从 Jobs-to-be-Done 视角提炼需求

功能 Job:
用户想完成什么任务?
-> 功能需求

情感 Job:
用户想获得什么感受?
-> 体验需求

社会 Job:
用户想展示什么形象?
-> 社交需求

示例:
功能 Job:高效管理项目进度 -> 项目管理功能
情感 Job:减少焦虑,感到掌控 -> 可视化进度、提醒功能
社会 Job:看起来专业、有条理 -> 专业界面、报告生成
```

**输出格式**:
```
需求洞察

显性需求:
| 需求 | 用户原话 | 优先级 |
|------|---------|--------|
| ... | ... | 高/中/低 |

隐性需求:
| 需求 | 推断依据 | 优先级 |
|------|---------|--------|
| ... | ... | 高/中/低 |

JTBD 需求:
- 功能 Job:[需求]
- 情感 Job:[需求]
- 社会 Job:[需求]
```

### Step 6: 制定行动建议 (15-20 分钟)

**建议制定方法**:

#### 6.1 产品改进建议
```
基于需求洞察提出产品改进建议

示例:
洞察:用户需要智能搜索文件
建议:增加全文搜索和标签功能

洞察:用户需要自动保存
建议:增加自动保存和版本管理功能

洞察:用户需要简单操作界面
建议:优化界面设计,减少学习成本
```

#### 6.2 验证计划建议
```
基于洞察提出需要验证的假设

示例:
洞察:用户说愿意为高级功能付费
验证:设计预售测试验证支付意愿

洞察:用户表示会使用新功能
验证:设计假门测试验证使用意愿
```

#### 6.3 后续访谈建议
```
基于本次访谈提出后续访谈方向

示例:
发现:用户决策逻辑出乎意料
后续:深入访谈更多用户验证决策模式

发现:用户有未预期使用场景
后续:访谈更多用户探索场景多样性
```

**输出格式**:
```
行动建议

产品改进建议:
| 洞察 | 建议 | 优先级 | 负责人 |
|------|------|--------|--------|
| ... | ... | P0/P1/P2 | ... |

验证计划建议:
| 洞察 | 假设 | 验证方式 | 时间 |
|------|------|---------|------|
| ... | ... | ... | ... |

后续访谈建议:
| 发现 | 后续方向 | 计划时间 |
|------|---------|---------|
| ... | ... | ... |
```

## 常见错误

- 只记录用户观点,不追问事实
- 忽视用户行为的矛盾性
- 过度解读用户原话
- 没有提取具体可执行的建议
- 忽视情感层面的洞察
- 只关注痛点,忽视现有解决方案的价值

## 与其他方法的关系

- 前置方法: interview-script(访谈脚本)
- 并行方法: 多次访谈综合分析
- 后置方法: 用户画像、产品需求提炼
