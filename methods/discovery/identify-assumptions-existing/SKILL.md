# Identify Assumptions - Existing Product (现有产品假设识别)

## 目的

针对现有产品的功能想法，系统化识别隐藏的假设和风险，避免基于未经验证的假设投入大量开发资源，确保决策基于证据而非猜测。

## 核心概念

```
假设识别四象限
  │
  ├─ 价值假设 (Value)
  │   └─ 用户真的需要这个功能吗？
  │
  ├─ 可用性假设 (Usability)
  │   └─ 用户能发现并理解如何使用吗？
  │
  ├─ 可行性假设 (Feasibility)
  │   └─ 我们能实现吗？
  │
  └─ 商业可行性假设 (Viability)
      └─ 能为业务带来正向回报吗？
```

## 何时使用

- ✅ 确定要开发某个功能前，识别风险
- ✅ 功能想法较多，需要验证优先级
- ✅ 团队对功能价值有分歧
- ✅ 需要向利益相关者解释为什么验证
- ✅ 制定实验计划前

## 使用步骤

### Step 1: 准备阶段 - 明确功能想法

**输入材料**:
1. **功能描述**: 这个功能是什么？解决什么问题？
2. **目标用户**: 为谁设计？
3. **预期效果**: 期望带来什么改变？

**准备问题**:
```
- 这个功能想解决什么问题？
- 为什么认为这是一个好主意？
- 如果我们错了，最可能错在哪里？
- 我们在哪些地方在做假设？
```

**输出格式**:
```
## 功能想法明确

- 功能名称：[名称]
- 功能描述：[一句话描述]
- 目标用户：[用户类型]
- 预期效果：[预期的业务影响]
```

### Step 2: 价值假设识别 (20-30 分钟)

**核心问题**: 用户真的需要这个功能吗？

**识别方法**:

#### 2.1 问题假设
```
问题：我们认为用户有这个问题，但真的有吗？

示例：
功能：增加「一键分享到朋友圈」
假设：用户想分享商品到朋友圈
验证方式：
- 查看现有分享数据
- 用户访谈询问分享意愿
- 竞品是否有类似功能
```

#### 2.2 解决方案假设
```
问题：我们认为这个方案能解决问题，真的能吗？

示例：
功能：增加搜索历史记录
假设：用户会通过搜索历史重新找到商品
验证方式：
- 用户访谈：用户如何找回之前看过的商品？
- 数据分析：有多少用户重复搜索相同关键词
```

#### 2.3 优先级假设
```
问题：我们认为这个问题足够重要，真的吗？

示例：
功能：优化搜索结果排序算法
假设：搜索体验是影响用户流失的关键因素
验证方式：
- 数据分析：搜索与转化的相关性
- 用户反馈：搜索相关的投诉占比
```

**输出格式**:
```
## 价值假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| VAL-01 | 用户有 X 需求 | 高/中/低 | ... | ... |
| VAL-02 | 这个方案能解决 X | 高/中/低 | ... | ... |
```

### Step 3: 可用性假设识别 (20-30 分钟)

**核心问题**: 用户能发现并理解如何使用吗？

**识别方法**:

#### 3.1 可发现性假设
```
问题：用户能找到这个功能吗？

示例：
功能：新增「收藏夹」功能
假设：用户能找到收藏入口
验证方式：
- 用户测试：观察用户是否找到收藏按钮
- 数据分析：收藏功能的使用率
```

#### 3.2 理解性假设
```
问题：用户能理解这个功能的作用吗？

示例：
功能：新增「智能推荐」功能
假设：用户理解为什么推荐这些商品
验证方式：
- 用户测试：询问用户对推荐的理解
- A/B 测试：不同文案的点击率
```

#### 3.3 操作性假设
```
问题：用户能顺利操作完成吗？

示例：
功能：新增商品对比功能
假设：用户能理解如何选择和对比商品
验证方式：
- 用户测试：观察用户操作流程
- 错误率分析：用户操作错误次数
```

**输出格式**:
```
## 可用性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| USA-01 | 用户能找到 X 入口 | 高/中/低 | ... | ... |
| USA-02 | 用户能理解 X 的作用 | 高/中/低 | ... | ... |
```

### Step 4: 可行性假设识别 (20-30 分钟)

**核心问题**: 我们能实现吗？

**识别方法**:

#### 4.1 技术可行性
```
问题：技术上能实现吗？

示例：
功能：实时视频客服
假设：我们有能力实现低延迟的视频通话
验证方式：
- 技术调研：现有技术方案评估
- 技术预研：小规模技术验证
```

#### 4.2 资源可行性
```
问题：有足够的资源吗？

示例：
功能：个性化推荐系统
假设：团队有足够的时间和人力开发
验证方式：
- 工期估算：开发工作量评估
- 人力盘点：团队可用资源
```

#### 4.3 时间可行性
```
问题：能在预期时间内完成吗？

示例：
功能：双 11 大促活动
假设：能在 2 个月内完成开发
验证方式：
- 排期规划：详细开发计划
- 风险评估：潜在延期风险
```

**输出格式**:
```
## 可行性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| FEA-01 | 技术上能实现 X | 高/中/低 | ... | ... |
| FEA-02 | 有足够资源开发 X | 高/中/低 | ... | ... |
```

### Step 5: 商业可行性假设识别 (20-30 分钟)

**核心问题**: 能为业务带来正向回报吗？

**识别方法**:

#### 5.1 商业价值假设
```
问题：这个功能能带来商业回报吗？

示例：
功能：会员专属客服
假设：能提升会员续费率，抵消客服成本
验证方式：
- 财务测算：成本收益分析
- 竞品调研：同类功能的商业效果
```

#### 5.2 战略一致性假设
```
问题：与产品战略方向一致吗？

示例：
功能：社交功能模块
假设：社交能增强用户粘性，符合战略方向
验证方式：
- 战略回顾：与战略目标对标
- 数据预测：对关键指标的影响预测
```

#### 5.3 风险评估
```
问题：有潜在负面影响吗？

示例：
功能：增加广告位
假设：不会显著影响用户体验和留存
验证方式：
- 竞品调研：竞品广告密度和效果
- 小范围测试：灰度发布测试影响
```

**输出格式**:
```
## 商业可行性假设清单

| 编号 | 假设描述 | 风险等级 | 验证方式 | 现有证据 |
|------|---------|---------|---------|---------|
| VIA-01 | X 能带来 Y 收益 | 高/中/低 | ... | ... |
| VIA-02 | X 不会有负面影响 | 高/中/低 | ... | ... |
```

### Step 6: 整合假设清单 (20-30 分钟)

**整合方法**:
1. 将所有假设汇总到一张表
2. 标注假设间的依赖关系
3. 评估整体风险等级

**假设依赖关系图**:
```
功能想法
  │
  ├─ 价值假设 1 (用户有需求)
  │   ├─ 可用性假设 1 (能找到入口)
  │   └─ 商业假设 1 (有价值回报)
  │
  └─ 可行性假设 1 (能实现)
      └─ 可行性假设 2 (有资源)
```

**输出格式**:
```
## 功能假设总表

| 假设ID | 类型 | 描述 | 风险等级 | 验证方式 | 优先级 |
|--------|------|------|---------|---------|--------|
| VAL-01 | 价值 | ... | 高 | ... | P0 |
| USA-01 | 可用性 | ... | 中 | ... | P1 |
| FEA-01 | 可行性 | ... | 低 | ... | P2 |
| VIA-01 | 商业 | ... | 高 | ... | P0 |

## 高风险假设（必须验证）
...

## 中等风险假设（建议验证）
...

## 低风险假设（可跳过验证）
...
```

## 常见错误

- ❌ 只关注价值假设，忽视可用性和可行性
- ❌ 假设描述过于笼统，无法验证
- ❌ 过度乐观，低估风险等级
- ❌ 没有区分假设优先级
- ❌ 忽视假设间的依赖关系
