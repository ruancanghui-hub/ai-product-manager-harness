# User Personas (用户画像)

## 目的

基于真实用户研究数据创建详细的用户画像,帮助团队深入理解目标用户,形成共识,指导产品设计、开发和营销决策。

## 核心概念

```
用户画像结构
  |
  +-- 基本信息
  |    +-- 人口统计、职业、家庭
  |
  +-- 背景故事
  |    +-- 生活状态、工作环境
  |
  +-- 目标动机
  |    +-- 功能目标、情感目标
  |
  +-- 行为特征
  |    +-- 使用习惯、技能水平
  |
  +-- 痛点需求
  |    +-- 当前痛点、期望解决方案
  |
  +-- 使用场景
  |    +-- 典型场景、触发条件
  |
  +-- 决策因素
       +-- 决策逻辑、影响因素
```

## 何时使用

- 产品发现阶段,定义目标用户
- 设计阶段,指导产品设计
- 开发阶段,帮助团队理解用户
- 营销阶段,指导营销策略
- 团队对用户理解有分歧时

## 使用步骤

### Step 1: 准备阶段 - 收集用户研究数据

**输入材料**:
1. 用户访谈记录(至少 5-10 人)
2. 用户问卷调查数据
3. 用户行为数据分析
4. 客户反馈和支持工单

**准备工作**:
```
- 汇总所有用户研究数据
- 识别用户特征模式
- 提取共同行为特征
- 整理典型用户故事
```

**输出格式**:
```
用户研究数据汇总

数据来源:
- 用户访谈:10 人,时长 45 分钟/人
- 问卷调查:200 人,问卷长度 20 题
- 行为数据:过去 3 个月用户行为数据
- 客户反馈:过去 6 个月客服工单 500 条

数据质量评估:
- 覆盖度:[覆盖的用户群体]
- 代表性:[是否代表目标用户]
- 深度:[数据详细程度]
```

### Step 2: 识别用户分群 (30-45 分钟)

**分群方法**:

#### 2.1 行为分群
```
按用户行为特征分群

分群维度:
- 使用频率:高频/中频/低频
- 使用场景:工作/生活/娱乐
- 功能使用:核心功能/辅助功能
- 技能水平:专家/普通/新手

示例:
高频用户:每天使用,依赖核心功能
中频用户:每周使用,偶尔使用辅助功能
低频用户:每月使用,功能了解有限
```

#### 2.2 目标分群
```
按用户目标和动机分群

分群维度:
- 主要目标:效率提升/成本降低/体验改善
- 使用动机:工作需要/个人兴趣/社交需求
- 期望价值:功能价值/情感价值/社交价值

示例:
效率型用户:追求效率提升,使用核心功能
体验型用户:追求体验改善,重视界面设计
社交型用户:追求社交需求,关注分享功能
```

#### 2.3 痛点分群
```
按用户痛点分群

分群维度:
- 主要痛点:时间压力/成本压力/技能不足
- 痛点程度:高痛点/中痛点/低痛点
- 解决迫切性:迫切解决/可以等待/不太重要

示例:
高痛点用户:痛点强烈,迫切需要解决方案
中痛点用户:痛点存在,愿意寻找解决方案
低痛点用户:痛点较弱,对解决方案需求不强
```

**输出格式**:
```
用户分群结果

分群 1:高频效率型用户
- 特征:每天使用,追求效率,痛点强烈
- 占比:约 30%
- 代表性:高

分群 2:中频体验型用户
- 特征:每周使用,重视体验,痛点中等
- 占比:约 50%
- 代表性:高

分群 3:低频社交型用户
- 特征:偶尔使用,关注社交,痛点较弱
- 占比:约 20%
- 代表性:中

确定画像数量:
- 建议 3-5 个主要画像
- 覆盖 80% 以上用户
```

### Step 3: 创建用户画像 (60-90 分钟)

**画像内容框架**:

#### 3.1 基本信息
```
内容:
- 姓名:虚构但真实的姓名
- 年龄:年龄范围
- 性别:性别
- 职业:具体职业
- 公司:公司类型和规模
- 家庭:家庭状况
- 收入:收入水平
- 地域:居住地区

示例:
姓名:李明
年龄:32 岁
性别:男
职业:项目经理
公司:中型科技公司,50-200 人
家庭:已婚,有一个 3 岁孩子
收入:月薪 2-3 万
地域:北京
```

#### 3.2 背景故事
```
内容:
- 生活状态:日常生活描述
- 工作环境:工作场景描述
- 技术水平:技术能力评估
- 个性特征:性格特点描述

示例:
背景故事:
李明是一名项目经理,负责多个软件开发项目。每天工作繁忙,需要协调团队成员、跟踪项目进度、汇报给上级。他工作认真负责,但经常感到时间不够用。他熟悉基本软件操作,但对复杂工具学习成本有顾虑。性格务实,重视效率,不喜欢花时间学习复杂功能。

技术水平:
熟练使用 Excel、PPT,会使用基础项目管理工具,对新技术接受度中等。

个性特征:
务实、高效、不喜欢复杂、重视结果。
```

#### 3.3 目标动机
```
内容:
- 功能目标:想完成什么任务
- 情感目标:想获得什么感受
- 社会目标:想展示什么形象
- 使用动机:为什么使用产品

示例:
功能目标:
- 高效管理多个项目进度
- 快速分配任务给团队成员
- 自动生成项目报告
- 及时发现问题并提醒

情感目标:
- 减少焦虑,感到掌控
- 工作完成后有成就感
- 不再担心遗漏重要事项

社会目标:
- 在团队中显得专业、有条理
- 给上级留下好印象

使用动机:
- 工作需要,不得不管理项目
- 希望减少工作量,提高效率
```

#### 3.4 行为特征
```
内容:
- 使用频率:多久使用一次
- 使用时间:什么时候使用
- 使用场景:在什么场景下使用
- 核心功能:经常使用哪些功能
- 操作习惯:有什么操作偏好
- 技能水平:熟练程度如何

示例:
使用频率:每天多次使用,早上、中午、下班前
使用时间:工作时间内,主要是项目协调时
使用场景:
- 早上:查看今日任务,分配工作
- 中午:跟踪进度,处理问题
- 下班前:整理进度,准备汇报

核心功能:
- 任务分配和跟踪
- 进度可视化
- 报告生成
- 提醒通知

操作习惯:
- 喜欢简单直观的操作
- 不愿意花时间学习复杂功能
- 习惯用快捷键和模板

技能水平:
- 熟练使用核心功能
- 较少使用高级功能
- 遇到问题会求助同事
```

#### 3.5 痛点需求
```
内容:
- 当前痛点:最大的困扰是什么
- 现有方案:现在如何解决
- 方案不足:现有方案的问题
- 期望方案:希望什么解决方案
- 支付意愿:愿意为解决方案付费吗

示例:
当前痛点:
- 手动更新进度耗时易错
- 多项目难以统筹管理
- 信息分散,难以追踪
- 报告生成耗时重复

现有方案:
用 Excel 手动记录每个项目进度,每周更新一次,手工整理成报告。

方案不足:
- Excel 容易出错,版本混乱
- 多项目难以统筹,需要多个表格
- 信息分散,查找困难
- 报告手工整理,耗时重复

期望方案:
- 一键更新进度,自动同步
- 多项目统一管理,一目了然
- 信息集中存储,快速查找
- 自动生成报告,节省时间

支付意愿:
愿意为项目管理工具付费,但价格要合理(每年不超过 1000 元)。
```

#### 3.6 使用场景
```
内容:
- 场景描述:具体使用场景
- 触发条件:什么触发使用
- 使用流程:使用过程描述
- 完成标准:如何判断完成

示例:
场景 1:早会前查看今日任务
场景描述:每天早会前 10 分钟,李明打开工具查看今日任务
触发条件:早会提醒,需要准备今日安排
使用流程:
1. 打开工具,查看今日待办
2. 检查各项目进度
3. 确认今日重点任务
4. 分配任务给团队成员
完成标准:清楚今日安排,团队已收到任务

场景 2:每周进度汇报
场景描述:每周五下班前,整理本周进度汇报给上级
触发条件:周五汇报提醒
使用流程:
1. 查看各项目本周进展
2. 标记完成和延期任务
3. 生成项目报告
4. 发送给上级
完成标准:报告已发送,内容准确完整

场景 3:项目问题处理
场景描述:发现项目延期或问题,需要及时处理
触发条件:项目延期提醒或团队成员反馈
使用流程:
1. 收到延期提醒
2. 查看延期任务详情
3. 分析原因和影响
4. 协调资源或调整计划
完成标准:问题已处理,团队已收到通知
```

#### 3.7 决策因素
```
内容:
- 决策触发:什么触发购买决策
- 关键因素:决策时考虑哪些因素
- 决策障碍:什么阻碍决策
- 信任来源:信任什么信息来源

示例:
决策触发:
- 项目延期频繁,上级不满
- 团队协作混乱,效率低下
- 同事推荐好用的工具

关键因素(按重要性排序):
1. 易用性:学习成本低,快速上手
2. 功能匹配:能满足核心需求
3. 价格合理:不超过预算
4. 团队接受:团队成员愿意使用
5. 数据安全:数据不会丢失泄露

决策障碍:
- 学习成本高,担心团队不接受
- 价格超出预算
- 不确定是否真的能解决问题
- 迁移现有数据的成本

信任来源:
- 同事推荐和使用经验
- 产品试用体验
- 用户评价和案例
- 权威媒体评测
```

**输出格式**:
```
用户画像:[画像名称]

基本信息:
- 姓名:[虚构姓名]
- 年龄:[年龄]
- ...

背景故事:
[详细故事描述]

目标动机:
- 功能目标:[目标列表]
- 情感目标:[目标列表]
- ...

行为特征:
- 使用频率:[频率]
- 使用场景:[场景列表]
- ...

痛点需求:
- 当前痛点:[痛点列表]
- 期望方案:[期望描述]
- ...

使用场景:
场景 1:[场景描述]
场景 2:[场景描述]
...

决策因素:
- 关键因素:[因素列表]
- 决策障碍:[障碍列表]
- ...
```

### Step 4: 完善画像细节 (30-45 分钟)

**细节补充**:

#### 4.1 添加真实照片或插画
```
目的:让画像更加真实可信

方法:
- 使用 AI 生成的人物照片
- 使用插画风格的人物形象
- 避免过于完美或不真实的形象

建议:
照片应该反映画像的特征
- 年龄、性别、职业相符
- 服装、环境符合画像描述
- 表情符合画像性格特征
```

#### 4.2 添加典型语录
```
目的:用用户原话增强真实感

示例:
李明的典型语录:
- "每天都要花半小时更新进度,太浪费时间了"
- "我就想要一个简单好用的工具,别搞得那么复杂"
- "能自动生成报告就好了,现在每周都要手工整理"
- "价格不能太贵,不然老板不会批准"

选择标准:
- 来自真实用户访谈
- 反映画像特征
- 生动有说服力
```

#### 4.3 绘制画像卡片
```
目的:创建简洁的画像总结卡片

内容:
- 照片/插画
- 姓名和标签
- 关键特征(3-5 条)
- 核心痛点(1-2 条)
- 主要需求(1-2 条)
- 典型语录(1 条)

格式:
一页纸或一张卡片,简洁直观
```

**输出格式**:
```
画像细节补充

照片:[照片或插画链接]

典型语录:
- "[语录 1]"
- "[语录 2]"
- ...

画像卡片:
[一页纸总结]
```

### Step 5: 验证画像准确性 (30-45 分钟)

**验证方法**:

#### 5.1 内部验证
```
团队成员验证画像准确性

验证问题:
- 这个画像符合我们接触的真实用户吗?
- 画像描述的行为特征是否真实?
- 画像的痛点和需求是否准确?

验证方式:
- 团队讨论会
- 用户研究人员确认
- 客服团队确认
```

#### 5.2 用户验证
```
真实用户验证画像准确性

验证问题:
- 这个画像描述的人和你相似吗?
- 画像描述的特征符合你的情况吗?
- 画像的痛点你也有吗?

验证方式:
- 用户访谈反馈
- 问卷调查验证
```

**输出格式**:
```
画像验证结果

内部验证:
- 团队认同度:高/中/低
- 修改建议:[建议]

用户验证:
- 用户认同度:高/中/低
- 修改建议:[建议]

最终确认:
画像是否准确反映真实用户
```

## 常见错误

- 基于假设而非真实数据创建画像
- 画像过于笼统,缺乏细节
- 画像数量过多,失去焦点
- 画像过于完美,不真实
- 画像描述没有行为证据
- 忽视画像的验证和更新

## 与其他方法的关系

- 前置方法: 用户访谈、问卷调查
- 并行方法: customer-journey-map(客户旅程地图)
- 后置方法: 产品设计、营销策略
