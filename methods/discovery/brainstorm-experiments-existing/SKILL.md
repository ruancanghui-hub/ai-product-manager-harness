# Brainstorm Experiments - Existing Product (现有产品实验设计)

## 目的

针对现有产品的关键假设，设计低成本、快速的验证实验，在不投入大量开发资源的情况下，获取真实用户反馈和数据，降低产品决策风险。

## 核心概念

```
实验类型金字塔
  │
  ├─ 原型测试（最低成本）
  │   └─ Figma/InVision 原型、纸面原型
  │
  ├─ 假门测试（低成本）
  │   └─ 只实现 UI，点击后提示「开发中」
  │
  ├─ 烟雾测试（中成本）
  │   └─ 落地页、广告投放、应用商店截图
  │
  ├─ A/B 测试（中高成本）
  │   └─ 小流量灰度测试
  │
  └─ 技术预研（高成本）
      └─ 技术可行性验证
```

## 何时使用

- ✅ 识别出关键假设，需要设计验证方式
- ✅ 不确定某个功能是否值得开发
- ✅ 需要快速验证多个方案
- ✅ 资源有限，需要低成本验证
- ✅ 向利益相关者展示验证计划

## 使用步骤

### Step 1: 准备阶段 - 明确验证目标

**输入材料**:
1. **假设清单**: 从 prioritize-assumptions 得到的优先级假设
2. **假设描述**: 每个假设的具体内容
3. **验证预算**: 可用的时间、金钱、人力

**准备问题**:
```
- 我们要验证哪些假设？
- 每个假设的最小验证成本是多少？
- 我们有验证所需的资源吗？
- 如何判断验证成功或失败？
```

**输出格式**:
```
## 验证目标明确

- 优先验证假设：[假设ID 和描述]
- 验证预算：时间 X 周，预算 Y 元
- 成功标准：[明确的成功指标]
```

### Step 2: 选择实验类型 (20-30 分钟)

**实验类型决策树**:

#### 2.1 需求验证类实验
```
假设：用户有某个需求/问题

实验类型：
├─ 用户访谈（深度理解需求）
├─ 问卷调查（量化需求规模）
├─ 竞品分析（现有解决方案）
└─ 数据分析（现有用户行为）

示例：
假设：用户想在 APP 内分享商品到社交平台
验证方式：
- 数据分析：查看现有分享功能使用率
- 用户访谈：询问用户分享意愿和场景
- 竞品分析：竞品分享功能设计
```

#### 2.2 解决方案验证类实验
```
假设：某个方案能解决问题

实验类型：
├─ 原型测试（用户是否能理解和操作）
├─ 假门测试（用户是否愿意点击）
├─ 烟雾测试（用户是否有兴趣）
└─ A/B 测试（不同方案的优劣）

示例：
假设：增加「一键分享」能提升分享率
验证方式：
- 假门测试：增加按钮，点击后提示「开发中」
- 原型测试：Figma 原型测试用户操作
- A/B 测试：对比有无按钮的分享率
```

#### 2.3 价值验证类实验
```
假设：用户愿意为某个功能付费

实验类型：
├─ 预售测试（是否有人预付）
├─ 价格测试（愿意付多少）
├─ 众筹测试（是否有足够支持）
└─ 落地页测试（转化率）

示例：
假设：用户愿意为高级功能付费
验证方式：
- 预售测试：推出预售页面，看是否有订单
- 价格测试：不同价格点的转化率
```

#### 2.4 可行性验证类实验
```
假设：技术上能实现

实验类型：
├─ 技术预研（小规模技术验证）
├─ 技术调研（现有技术方案评估）
├─ 专家咨询（领域专家意见）
└─ 竞品对标（竞品已实现）

示例：
假设：我们能实现低延迟视频通话
验证方式：
- 技术预研：搭建 Demo 测试延迟
- 技术调研：评估第三方 SDK
- 专家咨询：音视频专家意见
```

**输出格式**:
```
## 实验类型选择

| 假设ID | 假设描述 | 推荐实验类型 | 次选实验类型 |
|--------|---------|-------------|-------------|
| VAL-01 | 用户有 X 需求 | 用户访谈 | 数据分析 |
| VAL-02 | 这个方案能解决 | 原型测试 | 假门测试 |
| VIA-01 | 用户愿意付费 | 预售测试 | 价格测试 |
| FEA-01 | 技术能实现 | 技术预研 | 专家咨询 |
```

### Step 3: 设计具体实验 (45-60 分钟)

**实验设计框架**:

#### 3.1 用户访谈实验
```
实验设计模板：

实验名称：[假设ID] - 用户访谈
验证假设：用户有 X 需求
目标用户：[用户画像]
样本量：5-10 人
访谈方式：面对面/视频/电话
访谈时长：30-45 分钟

访谈提纲：
1. 暖场：了解用户背景
2. 现状：用户现在如何解决这个问题
3. 痛点：用户遇到的最大困难
4. 愿望：用户期望的解决方案
5. 反馈：对假设方案的反应

成功标准：
- 70% 以上用户确认有此需求
- 用户能描述具体场景
- 用户表达解决问题的意愿

资源需求：
- 时间：1 周（招募 + 访谈 + 分析）
- 金钱：1000 元（用户激励）
- 人力：PM 2 天

执行计划：
D1-D2：招募用户
D3-D4：进行访谈
D5：整理分析
```

#### 3.2 原型测试实验
```
实验设计模板：

实验名称：[假设ID] - 原型测试
验证假设：用户能理解并使用这个功能
原型类型：Figma 可交互原型 / 纸面原型
测试用户：5-8 人
测试时长：30 分钟

测试流程：
1. 简单介绍背景（不提示解决方案）
2. 给用户一个任务
3. 观察用户操作过程
4. 询问用户的理解和感受
5. 收集改进建议

观察重点：
- 用户是否能独立完成任务
- 用户在哪些环节遇到困难
- 用户的情绪反应

成功标准：
- 80% 用户能独立完成任务
- 任务完成时间 < 3 分钟
- 用户满意度 > 4 分（5 分制）

资源需求：
- 时间：1 周
- 金钱：500 元（原型工具 + 用户激励）
- 人力：设计师 2 天 + PM 1 天

执行计划：
D1-D2：制作原型
D3-D4：用户测试
D5：整理分析
```

#### 3.3 假门测试实验
```
实验设计模板：

实验名称：[假设ID] - 假门测试
验证假设：用户会对这个功能感兴趣
测试位置：[页面位置]
测试流量：10% 用户
测试时长：1-2 周

实现方式：
- 在页面添加功能入口（按钮/链接）
- 点击后弹出提示「功能开发中，敬请期待」
- 记录点击率和点击用户特征

数据指标：
- 点击率：点击人数 / 曝光人数
- 点击用户特征：用户分群分析
- 重复点击率：是否多次点击

成功标准：
- 点击率 > 5%（或高于基准）
- 点击用户是目标用户群体
- 有用户反馈期待功能

资源需求：
- 时间：1 周（开发） + 1-2 周（数据收集）
- 金钱：0 元（内部开发）
- 人力：开发 1 天 + PM 1 天

执行计划：
W1：开发上线
W2-W3：数据收集
W4：数据分析
```

#### 3.4 A/B 测试实验
```
实验设计模板：

实验名称：[假设ID] - A/B 测试
验证假设：方案 A 比方案 B 效果更好
测试流量：10-20% 用户
测试时长：2-4 周

实验设计：
- 对照组（A）：[现有方案]
- 实验组（B）：[新方案]
- 分流方式：随机分流

数据指标：
- 主指标：[核心指标，如点击率、转化率]
- 辅助指标：[次要指标]
- 护栏指标：[不能恶化的指标]

样本量计算：
- 基准转化率：X%
- 期望提升：Y%
- 显著性水平：95%
- 所需样本：N 用户/组

成功标准：
- 实验组主指标显著优于对照组（p < 0.05）
- 护栏指标无显著恶化
- 提升幅度达到预期

资源需求：
- 时间：1 周（开发） + 2-4 周（测试）
- 金钱：0 元（内部开发）
- 人力：开发 2 天 + PM 2 天 + 数据分析 1 天

执行计划：
W1：开发上线
W2-W5：数据收集
W6：数据分析
```

#### 3.5 烟雾测试实验
```
实验设计模板：

实验名称：[假设ID] - 落地页测试
验证假设：用户对这个产品/功能感兴趣
测试渠道：广告投放 / 社交媒体 / SEO
测试时长：1-2 周

落地页设计：
- 标题：清晰的价值主张
- 内容：功能介绍、使用场景
- CTA：注册/预约/了解更多

数据指标：
- 页面浏览量（PV/UV）
- 注册/预约转化率
- 获客成本（CAC）

成功标准：
- 转化率 > X%（行业基准）
- 获客成本 < Y 元
- 用户注册后愿意接受后续联系

资源需求：
- 时间：1 周（页面制作） + 1-2 周（投放）
- 金钱：5000 元（广告投放）
- 人力：设计师 1 天 + 运营 2 天

执行计划：
W1：页面制作
W2-W3：广告投放
W4：数据分析
```

#### 3.6 预售测试实验
```
实验设计模板：

实验名称：[假设ID] - 预售测试
验证假设：用户愿意为这个功能付费
测试方式：预售页面 / 众筹
测试时长：2-4 周

预售设计：
- 产品描述：功能和价值
- 预售价格：早鸟价
- 交付时间：预计上线时间
- 退款承诺：如未上线全额退款

数据指标：
- 预售订单量
- 预售收入
- 用户反馈

成功标准：
- 预售订单量 > X 单
- 预售收入 > Y 元
- 用户反馈积极

资源需求：
- 时间：1 周（页面制作） + 2-4 周（预售）
- 金钱：1000 元（页面制作 + 宣传）
- 人力：PM 2 天 + 运营 2 天

执行计划：
W1：页面制作
W2-W5：预售宣传
W6：数据分析
```

#### 3.7 技术预研实验
```
实验设计模板：

实验名称：[假设ID] - 技术预研
验证假设：技术上能实现 X 功能
预研范围：[具体技术点]
预研时长：1-2 周

预研内容：
- 技术调研：现有技术方案
- 技术选型：不同方案对比
- 技术验证：搭建 Demo 测试
- 风险评估：技术风险点

评估维度：
- 技术可行性：能实现吗？
- 技术复杂度：难度多大？
- 性能表现：能满足性能要求吗？
- 技术成本：需要多少资源？

成功标准：
- Demo 能正常运行
- 性能满足要求
- 技术风险可控

资源需求：
- 时间：1-2 周
- 金钱：0 元（内部技术预研）
- 人力：工程师 1-2 周

执行计划：
W1：技术调研和选型
W2：Demo 开发和测试
```

**输出格式**:
```
## 实验设计详情

### 实验名称：[实验ID]
- 验证假设：[假设描述]
- 实验类型：[类型]
- 样本量：[人数/流量]
- 测试时长：[时间]
- 成功标准：[明确标准]
- 资源需求：[时间/金钱/人力]
- 执行计划：[详细计划]
```

### Step 4: 评估实验可行性 (20-30 分钟)

**评估维度**:

#### 4.1 技术可行性
```
问题：我们能实现这个实验吗？

评估：
- 需要哪些技术开发？
- 技术难度多大？
- 有现成工具支持吗？

示例：
实验：A/B 测试新功能
评估：
- 需要开发分流系统（已有 A/B 平台）
- 技术难度：低
- 可行性：高
```

#### 4.2 资源可行性
```
问题：有足够的资源吗？

评估：
- 时间是否充足？
- 预算是否充足？
- 人力是否充足？

示例：
实验：用户访谈 10 人
评估：
- 时间：1 周（充足）
- 预算：1000 元（充足）
- 人力：PM 2 天（充足）
- 可行性：高
```

#### 4.3 时间可行性
```
问题：能在预期时间内完成吗？

评估：
- 实验准备需要多长时间？
- 数据收集需要多长时间？
- 分析总结需要多长时间？

示例：
实验：A/B 测试 2 周
评估：
- 准备：1 周
- 收集：2 周
- 分析：1 周
- 总计：4 周
- 可行性：可接受
```

**输出格式**:
```
## 实验可行性评估

| 实验ID | 实验名称 | 技术可行性 | 资源可行性 | 时间可行性 | 综合评估 |
|--------|---------|-----------|-----------|-----------|---------|
| EXP-01 | 用户访谈 | 高 | 高 | 高 | ✅ 可执行 |
| EXP-02 | A/B 测试 | 高 | 中 | 中 | ⚠️ 需调整 |
| EXP-03 | 预售测试 | 中 | 中 | 低 | ❌ 不可行 |
```

### Step 5: 制定实验计划 (20-30 分钟)

**计划内容**:

#### 5.1 实验优先级
```
实验优先级 = 假设优先级 × 实验可行性

示例：
实验 A：假设 P0 × 可行性高 = 最高优先级
实验 B：假设 P1 × 可行性高 = 高优先级
实验 C：假设 P0 × 可行性低 = 中优先级（寻找替代方案）
```

#### 5.2 实验时间表
```
## 实验时间表

| 周次 | 实验名称 | 实验类型 | 负责人 | 状态 |
|------|---------|---------|--------|------|
| W1 | 用户访谈 | 访谈 | PM | 计划中 |
| W1-W2 | 原型测试 | 原型 | 设计师 | 计划中 |
| W2-W5 | A/B 测试 | A/B | PM + 开发 | 计划中 |
| W3-W4 | 数据分析 | 分析 | 数据分析师 | 计划中 |
```

#### 5.3 资源分配
```
## 资源分配

| 实验 | 时间 | 预算 | 人力 | 来源 |
|------|------|------|------|------|
| 用户访谈 | 1 周 | 1000 元 | PM 2 天 | 运营预算 |
| 原型测试 | 1 周 | 500 元 | 设计师 2 天 | 设计预算 |
| A/B 测试 | 4 周 | 0 元 | 开发 2 天 + PM 2 天 | 开发资源 |
```

## 常见错误

- ❌ 实验设计过于复杂，成本过高
- ❌ 成功标准不明确，无法判断结果
- ❌ 样本量不足，结果不可靠
- ❌ 忽视护栏指标，导致负面影响
- ❌ 实验时间过短，数据不充分
- ❌ 同时测试多个变量，无法归因

## 与其他方法的关系

- **前置方法**: prioritize-assumptions（假设排序）
- **并行方法**: 无
- **后置方法**: 执行实验 → 分析结果 → 决策
