# Brainstorm Ideas - New Product (新产品创意发散)

## 目的

针对新产品或新业务线的初始发现阶段，从多角色视角系统化发散功能创意，确保全面覆盖用户需求、体验设计、技术可行性等维度，避免过早收敛。

## 核心概念

```
新产品功能发散
  │
  ├─ PM 视角
  │   ├─ 核心价值主张
  │   ├─ 用户关键任务
  │   └─ 差异化竞争力
  │
  ├─ 设计师视角
  │   ├─ 首次使用体验
  │   ├─ 情感化设计
  │   └─ 品牌记忆点
  │
  └─ 工程师视角
      ├─ 技术可行性
      ├─ MVP 范围控制
      └─ 架构可扩展性
```

## 何时使用

- ✅ 新产品从 0 到 1 的初始规划
- ✅ 业务线扩展，需要定义核心功能
- ✅ 产品方向验证后，需要具体功能设计
- ✅ 团队对「做什么功能」有分歧
- ✅ 需要向投资者或利益相关者展示产品规划

## 使用步骤

### Step 1: 准备阶段 - 明确产品方向

**输入材料**:
1. **价值主张**: 产品要解决什么核心问题？
2. **目标用户**: 用户画像和核心场景
3. **竞品分析**: 现有解决方案和不足
4. **业务目标**: 成功指标和时间表

**准备问题**:
```
- 我们的核心价值主张是什么？
- 目标用户是谁？他们现在如何解决问题？
- 竞品的不足是什么？
- 6-12 个月后的成功是什么样子？
```

**输出格式**:
```
## 产品方向明确

- 核心价值主张：[一句话描述]
- 目标用户：[用户画像]
- 核心场景：[3-5 个关键场景]
- 成功指标：[6-12 个月目标]
```

### Step 2: PM 视角发散 (30-45 分钟)

**思维框架**:

#### 2.1 核心价值主张拆解
```
问题：如何实现核心价值主张？

方法：
1. 将价值主张拆解为具体功能需求
2. 识别最小可行功能集（MVP）
3. 确定功能优先级

示例：
价值主张：让用户快速找到并购买心仪商品
拆解：
- 搜索功能（关键词、筛选、排序）
- 商品详情（图片、价格、评价）
- 购买流程（加购、下单、支付）
- 订单管理（查询、退款、售后）
```

#### 2.2 用户关键任务分析
```
问题：用户完成核心任务需要哪些功能？

方法：
1. 列出用户关键任务清单
2. 为每个任务设计必要功能
3. 识别任务间的关联功能

示例：
关键任务 1：发现商品
- 推荐功能（个性化、热门、新品）
- 分类浏览
- 搜索功能

关键任务 2：决策购买
- 商品对比
- 评价查看
- 收藏功能
```

#### 2.3 差异化竞争力设计
```
问题：如何让产品在竞争中脱颖而出？

方法：
1. 分析竞品功能对标
2. 识别差异化机会点
3. 设计杀手级功能

示例：
竞品对标：
- 竞品 A 有 X 功能，我们需要吗？
- 竞品 B 缺少 Y 功能，我们可以做吗？

差异化机会：
- 更智能的推荐
- 更快的搜索
- 更好的售后服务
```

**输出格式**:
```
## PM 视角功能清单

| 编号 | 功能 | 对应价值主张 | 优先级 | MVP 必需 |
|------|------|-------------|--------|----------|
| PM-01 | ... | ... | P0/P1/P2 | 是/否 |
| PM-02 | ... | ... | P0/P1/P2 | 是/否 |
```

### Step 3: 设计师视角发散 (30-45 分钟)

**思维框架**:

#### 3.1 首次使用体验设计
```
问题：如何让新用户快速理解产品价值？

方法：
1. 设计「Aha Moment」路径
2. 规划新手引导策略
3. 避免空白状态

示例：
- 首次打开 App 的欢迎流程
- 账号注册的最简路径
- 首次使用的引导动画
- 空状态的内容填充
```

#### 3.2 情感化设计
```
问题：如何让用户喜欢我们的产品？

方法：
1. 设计品牌性格和语气
2. 增加微交互和动效
3. 创造惊喜时刻

示例：
- 加载等待时的趣味动画
- 完成任务后的成就感反馈
- 个性化文案和插图
```

#### 3.3 品牌记忆点设计
```
问题：如何让用户记住我们的产品？

方法：
1. 设计独特的视觉标识
2. 创造品牌专属元素
3. 保持跨平台一致性

示例：
- 独特的配色方案和 Logo
- 品牌吉祥物或插画风格
- 统一的文案风格
```

**输出格式**:
```
## 设计师视角功能清单

| 编号 | 功能/体验点 | 情感目标 | 品牌价值 | MVP 必需 |
|------|------------|---------|---------|----------|
| DES-01 | ... | ... | ... | 是/否 |
| DES-02 | ... | ... | ... | 是/否 |
```

### Step 4: 工程师视角发散 (30-45 分钟)

**思维框架**:

#### 4.1 技术可行性评估
```
问题：哪些功能在技术上可行？

方法：
1. 评估技术实现难度
2. 识别技术风险点
3. 确定技术选型

示例：
功能：实时商品推荐
技术方案：
- 客户端：推荐算法 SDK
- 服务端：机器学习模型
- 数据：用户行为采集
风险：冷启动问题、模型精度
```

#### 4.2 MVP 范围控制
```
问题：如何用最小成本验证价值？

方法：
1. 识别核心功能的最小实现
2. 去除非必要功能
3. 设计快速验证方案

示例：
完整版：AI 智能推荐 + 千人千面
MVP 版：热门商品推荐 + 简单规则推荐
验证目标：用户是否使用推荐功能
```

#### 4.3 架构可扩展性设计
```
问题：如何为未来发展预留空间？

方法：
1. 设计模块化架构
2. 预留接口和扩展点
3. 避免过度设计

示例：
- API 设计考虑未来多端
- 数据库设计考虑扩展字段
- 代码结构考虑模块化
```

**输出格式**:
```
## 工程师视角功能清单

| 编号 | 功能 | 技术可行性 | MVP 实现方案 | 架构考虑 |
|------|------|-----------|-------------|---------|
| ENG-01 | ... | 高/中/低 | ... | ... |
| ENG-02 | ... | 高/中/低 | ... | ... |
```

### Step 5: 整合与 MVP 定义 (45 分钟)

**整合方法**:
1. 将三个视角的想法汇总
2. 标注功能间的关联和依赖
3. 区分「必须有」「应该有」「可以有」

**MoSCoW 优先级法**:
```
- Must have（必须有）：没有这个功能，产品无法使用
- Should have（应该有）：没有这个功能，体验大幅下降
- Could have（可以有）：有更好，没有也可接受
- Won't have（暂不有）：明确不在 MVP 范围
```

**MVP 功能集定义**:
```
## MVP 功能清单

### 必须有（P0）
| 功能 | 描述 | 依赖功能 | 预计工期 |
|------|------|---------|---------|
| ... | ... | ... | ... |

### 应该有（P1）
| 功能 | 描述 | 依赖功能 | 预计工期 |
|------|------|---------|---------|
| ... | ... | ... | ... |

### 可以有（P2）
| 功能 | 描述 | 依赖功能 | 预计工期 |
|------|------|---------|---------|
| ... | ... | ... | ... |

### 暂不有（P3）
| 功能 | 描述 | 原因 | 后续规划 |
|------|------|------|---------|
| ... | ... | ... | ... |
```

## 常见错误

- ❌ 功能过多，MVP 范围失控
- ❌ 只考虑单一角色视角
- ❌ 忽视技术可行性评估
- ❌ 过度设计，延长上线时间
- ❌ 没有明确优先级标准
- ❌ 功能描述过于笼统

## 与其他方法的关系

- **前置方法**: value-proposition（定义价值主张）、user-personas（用户画像）
- **并行方法**: opportunity-solution-tree（可结合使用）
- **后置方法**: identify-assumptions（识别假设）、brainstorm-experiments-new（设计验证实验）
