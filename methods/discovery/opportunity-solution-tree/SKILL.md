# Opportunity Solution Tree (机会解决方案树)

## 目的

系统化探索产品机会空间，确保团队不会过早收敛到单一方案，而是先广度优先地探索所有可能的机会，再深度优先地验证最有价值的方案。

## 核心概念

```
成果 (Outcome)
  │
  ├─ 机会 1 (Opportunity)
  │   ├─ 方案 1.1 (Solution)
  │   ├─ 方案 1.2
  │   └─ 方案 1.3
  │
  ├─ 机会 2
  │   ├─ 方案 2.1
  │   └─ 方案 2.2
  │
  └─ 机会 3
      ├─ 方案 3.1
      └─ 方案 3.2
```

- **成果**: 你想实现的业务目标（如「提高用户留存」）
- **机会**: 用户需求、痛点或欲望（不是你的产品功能）
- **方案**: 解决机会的具体方式（这才涉及产品功能）

## 何时使用

- ✅ 新产品或新功能启动时，需求模糊
- ✅ 团队对「做什么」有分歧
- ✅ 想要确保没有错过重要的机会空间
- ✅ 需要向利益相关者解释「为什么做这个」

## 使用步骤

### Step 1: 定义成果（Outcome）

**问题**: 我们想实现什么业务目标？

**格式**:
```
成果 = [动词] + [指标] + [时间范围]

示例：
- 3 个月内将 7 日留存率从 35% 提升到 45%
- 本季度将月活跃用户从 100 万增长到 120 万
- 6 个月内将付费转化率从 5% 提升到 8%
```

**检查点**:
- [ ] 成果是否可量化？
- [ ] 是否有明确的时间范围？
- [ ] 是否与业务目标对齐？

### Step 2: 发散机会（Diverge Opportunities）

**问题**: 用户有什么需求、痛点或欲望与此成果相关？

**方法**:
1. **用户访谈**: 与 5-10 个目标用户进行深度访谈
2. **数据分析**: 查看现有用户行为数据
3. **竞品分析**: 看竞品解决了哪些需求
4. **客户反馈**: 分析客服工单、App Store 评论

**格式**:
```
机会模板：
[用户类型] 在 [场景] 下遇到 [痛点/需求/欲望]

示例：
- 新用户在首次打开 APP 时不知道该做什么
- 活跃用户想和朋友分享喜欢的内容但找不到入口
- 付费用户觉得会员权益不够清晰
```

**检查点**:
- [ ] 至少识别出 10-15 个机会？
- [ ] 机会是从用户视角描述的，而不是产品功能？
- [ ] 机会足够具体，可后续验证？

### Step 3: 映射机会到成果

**问题**: 这些机会如何与成果关联？

**方法**:
1. 将机会写在便签上
2. 按对成果的影响程度分组（高/中/低）
3. 标注机会之间的依赖关系

**可视化**:
```
成果：3 个月内将 7 日留存率从 35% 提升到 45%

高影响机会：
├─ 新用户首次使用时感到困惑 → 流失
├─ 活跃用户找不到有趣的内容 → 逐渐沉默
└─ 用户不知道有社交功能 → 孤独感

中影响机会：
├─ 推送通知过于频繁 → 打扰用户
└─ 搜索功能不够智能 → 找不到想要的内容
```

### Step 4: 发散方案（Diverge Solutions）

**问题**: 对每个高影响机会，有哪些可能的解决方案？

**方法**:
1. 对每个高影响机会进行头脑风暴
2. 每个机会至少生成 3-5 个方案
3. 不要过早评估方案可行性

**格式**:
```
机会：新用户首次使用时感到困惑 → 流失

方案：
1. 新手引导流程（分步骤教学）
2. 个性化推荐（首次登录询问兴趣）
3. 社交化引导（推荐关注好友）
4. 演示视频（展示核心功能）
5. 默认内容填充（避免空白状态）
```

**检查点**:
- [ ] 每个机会有 3+ 个方案？
- [ ] 方案是否多样（不是同一思路的变体）？
- [ ] 是否有「疯狂」的想法（可能颠覆现有假设）？

### Step 5: 收敛方案（Converge Solutions）

**问题**: 哪些方案最值得验证？

**评估维度**:
1. **用户价值**: 用户真的需要吗？
2. **业务价值**: 能推动成果吗？
3. **技术可行性**: 能实现吗？
4. **时间成本**: 多久能验证？

**优先级矩阵**:
```
         高用户价值
              │
    优先验证  │   值得探索
              │
─────────────┼───────────── 低技术可行性
              │
    暂缓      │   快速胜利
              │
         低用户价值
    高技术可行性
```

### Step 6: 设计实验（Design Experiments）

**问题**: 如何用最小成本验证方案？

**实验类型**:
1. **假门测试**: 只实现 UI，点击后提示「功能开发中」
2. **落地页测试**: 制作功能介绍页面，看用户是否愿意注册
3. **原型测试**: 用 Figma 原型进行可用性测试
4. **竞品分析**: 看竞品同类功能的数据和用户反馈
5. **烟雾测试**: 在应用商店发布截图，看转化率

**格式**:
```
方案：新手引导流程

假设：新手引导能将新用户 7 日留存率提升 10%

实验：
- 类型：原型测试
- 方法：制作 Figma 原型，邀请 10 名新用户测试
- 成功标准：80% 用户能独立完成核心任务
- 时间：3 天
- 成本：设计师 2 天 + PM 1 天
```

## 输出模板

```markdown
# 机会解决方案树 - [产品/功能名称]

## 成果
[明确的业务目标]

## 机会空间

### 高影响机会
1. **[机会描述]**
   - 来源：[访谈/数据/反馈]
   - 影响：[对成果的预期影响]
   - 方案：
     1. [方案 1] - 优先级：[高/中/低]
     2. [方案 2] - 优先级：[高/中/低]

### 中影响机会
...

### 低影响机会
...

## 验证计划
| 方案 | 实验类型 | 成功标准 | 时间 | 负责人 |
|------|---------|---------|------|--------|
| ... | ... | ... | ... | ... |

## 决策
- 立即执行：[方案]
- 需要验证：[方案]
- 暂缓：[方案]
```

## 常见错误

- ❌ 机会写成产品功能（「需要一个搜索框」而不是「找不到想要的内容」）
- ❌ 只探索 1-2 个机会就急于确定方案
- ❌ 跳过实验阶段，直接全量开发
- ❌ 成果不够量化，无法验证
- ❌ 团队没有共识就进入方案阶段

## 与其他方法的关系

- **前置方法**: user-interview（发现机会）
- **后置方法**: prioritize-assumptions（确定验证顺序）
- **并行方法**: value-proposition（明确价值主张）
