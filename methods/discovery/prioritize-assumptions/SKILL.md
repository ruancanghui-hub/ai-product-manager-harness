# Prioritize Assumptions (假设优先级排序)

## 目的

对产品发现阶段识别出的假设进行优先级排序，确保团队优先验证最关键、最高风险的假设，避免在错误方向上浪费资源。

## 核心概念

```
假设优先级矩阵
  │
  ├─ 高风险 + 低验证成本 → 立即验证
  │
  ├─ 高风险 + 高验证成本 → 优先级高，寻找低成本验证方式
  │
  ├─ 低风险 + 低验证成本 → 穿插验证
  │
  └─ 低风险 + 高验证成本 → 暂缓验证
```

## 何时使用

- ✅ 识别出大量假设，不知道从哪个开始验证
- ✅ 资源有限，需要聚焦关键假设
- ✅ 制定实验计划前
- ✅ 团队对假设优先级有分歧
- ✅ 需要向利益相关者解释验证顺序

## 使用步骤

### Step 1: 准备阶段 - 整理假设清单

**输入材料**:
1. **假设清单**: 从 identify-assumptions 得到的假设列表
2. **假设描述**: 每个假设的具体内容
3. **初步风险评估**: 已识别的风险等级

**准备问题**:
```
- 我们有哪些假设？
- 哪些假设如果不成立，产品就会失败？
- 哪些假设最容易被验证？
- 假设之间有什么依赖关系？
```

**输出格式**:
```
## 假设清单汇总

| 假设ID | 类型 | 描述 | 初步风险等级 |
|--------|------|------|-------------|
| VAL-01 | 价值 | 用户有 X 需求 | 高 |
| VAL-02 | 价值 | 用户愿意为 X 付费 | 高 |
| USA-01 | 可用性 | 用户能理解 X | 中 |
| ... | ... | ... | ... |
```

### Step 2: 评估假设重要性 (30-45 分钟)

**评估维度**:

#### 2.1 失败影响程度
```
问题：如果这个假设不成立，对产品有多大影响？

评估标准：
- 5 分：假设不成立，产品完全失败（关键路径假设）
- 4 分：假设不成立，核心功能无法实现
- 3 分：假设不成立，需要大幅调整产品方向
- 2 分：假设不成立，需要局部调整
- 1 分：假设不成立，影响较小

示例：
假设：用户愿意为高级功能付费
影响：如果不成立，商业模式不成立
评分：5 分（关键路径假设）
```

#### 2.2 假设不确定性
```
问题：我们对这个假设有多大把握？

评估标准：
- 5 分：完全不确定，纯猜测
- 4 分：有一些间接证据，但不充分
- 3 分：有相关数据支持，但不直接
- 2 分：有较多证据支持
- 1 分：基本确定，有强证据

示例：
假设：用户愿意为高级功能付费
证据：竞品有付费用户，但用户访谈中付费意愿不一
评分：4 分（较不确定）
```

**优先级评分**:
```
优先级分数 = 失败影响程度 × 假设不确定性

示例：
假设 A：失败影响 5 分 × 不确定性 4 分 = 20 分
假设 B：失败影响 3 分 × 不确定性 3 分 = 9 分

优先级：A > B
```

**输出格式**:
```
## 假设重要性评估

| 假设ID | 描述 | 失败影响 | 不确定性 | 优先级分数 |
|--------|------|---------|---------|-----------|
| VAL-01 | 用户有 X 需求 | 5 | 4 | 20 |
| VAL-02 | 用户愿意付费 | 5 | 5 | 25 |
| USA-01 | 用户能理解 X | 3 | 3 | 9 |
| ... | ... | ... | ... | ... |
```

### Step 3: 评估验证成本 (30-45 分钟)

**评估维度**:

#### 3.1 时间成本
```
问题：验证这个假设需要多长时间？

评估标准：
- 1 天：快速调研、数据分析
- 1 周：用户访谈、原型测试
- 2-4 周：小规模实验
- 1-3 月：技术开发 + 实验
- 3+ 月：大规模实验

示例：
假设：用户愿意为高级功能付费
验证方式：预售测试
时间成本：2 周
```

#### 3.2 金钱成本
```
问题：验证这个假设需要多少预算？

评估标准：
- 0 元：现有数据分析、内部访谈
- < 1000 元：小规模问卷、简单原型
- 1000-5000 元：用户招募、工具费用
- 5000-20000 元：广告投放、大规模测试
- > 20000 元：技术开发、大规模推广

示例：
假设：用户愿意为高级功能付费
验证方式：落地页 + 广告投放
金钱成本：5000 元
```

#### 3.3 人力成本
```
问题：验证这个假设需要多少人力？

评估标准：
- 0.5 人天：数据分析、问卷设计
- 1-2 人天：用户访谈、原型测试
- 1 周：小规模实验设计执行
- 2-4 周：技术开发 + 实验
- 1+ 月：大规模开发和实验

示例：
假设：用户愿意为高级功能付费
验证方式：预售测试
人力成本：1 周
```

**综合验证成本**:
```
验证成本分数 = (时间成本 × 权重) + (金钱成本 × 权重) + (人力成本 × 权重)

简化评估：
- 低成本：能快速低成本验证
- 中成本：需要一定资源和时间
- 高成本：需要大量投入
```

**输出格式**:
```
## 验证成本评估

| 假设ID | 验证方式 | 时间成本 | 金钱成本 | 人力成本 | 综合成本 |
|--------|---------|---------|---------|---------|---------|
| VAL-01 | 用户访谈 | 1 周 | 1000 元 | 2 人天 | 中 |
| VAL-02 | 预售测试 | 2 周 | 5000 元 | 1 周 | 中 |
| USA-01 | 原型测试 | 1 周 | 500 元 | 1 人天 | 低 |
| ... | ... | ... | ... | ... | ... |
```

### Step 4: 分析假设依赖关系 (20-30 分钟)

**依赖关系类型**:

#### 4.1 串联依赖
```
假设 A → 假设 B → 假设 C

含义：如果假设 A 不成立，假设 B 和 C 都无需验证

示例：
假设 A：用户有 X 需求
假设 B：用户愿意为 X 付费
假设 C：我们能以低成本实现 X

依赖链：如果用户没有需求（A 不成立），付费意愿（B）和实现成本（C）都不重要
验证顺序：先验证 A，再验证 B 和 C
```

#### 4.2 并联依赖
```
假设 A ┐
假设 B ├→ 功能实现
假设 C ┘

含义：多个假设同时成立才能成功

示例：
假设 A：用户能发现功能
假设 B：用户能理解功能
假设 C：用户愿意使用功能

关系：三个假设都需要成立，但可以并行验证
```

**依赖关系图绘制**:
```
## 假设依赖关系图

关键路径：
市场存在 → 用户有需求 → 用户愿意付费 → 能盈利

支线路径：
有需求 → 能理解产品 → 能使用产品

并行路径：
有需求 + 有能力实现 + 有资源支撑
```

**输出格式**:
```
## 假设依赖关系

| 假设ID | 依赖于 | 影响到 | 验证顺序 |
|--------|--------|--------|---------|
| VAL-01 | 无 | VAL-02, FEA-01 | 第 1 批 |
| VAL-02 | VAL-01 | VIA-01 | 第 2 批 |
| FEA-01 | VAL-01 | 无 | 第 2 批 |
| ... | ... | ... | ... |
```

### Step 5: 综合排序 (30-45 分钟)

**排序方法**:

#### 5.1 优先级矩阵
```
         高验证成本
              │
   战略优先   │   尽快验证
              │
──────────────┼────────────── 高重要性
              │
   暂缓验证   │   快速胜利
              │
         低验证成本
    低重要性

- 尽快验证（高重要性 + 低成本）：优先级最高
- 快速胜利（低重要性 + 低成本）：穿插验证
- 战略优先（高重要性 + 高成本）：寻找低成本验证方式
- 暂缓验证（低重要性 + 高成本）：最后考虑
```

#### 5.2 RICE 评分法
```
RICE 分数 = (Reach × Impact × Confidence) / Effort

- Reach：影响范围（多少人受影响）
- Impact：影响程度（1-5 分）
- Confidence：信心程度（1-5 分）
- Effort：验证成本（人月）

示例：
假设 A：Reach 10000 × Impact 5 × Confidence 4 / Effort 0.5 = 400000
假设 B：Reach 1000 × Impact 3 × Confidence 3 / Effort 0.25 = 36000

优先级：A > B
```

**输出格式**:
```
## 假设优先级总表

| 假设ID | 描述 | 重要性分数 | 验证成本 | 依赖顺序 | 最终优先级 |
|--------|------|-----------|---------|---------|-----------|
| VAL-01 | 用户有 X 需求 | 20 | 中 | 第 1 批 | P0 |
| VAL-02 | 用户愿意付费 | 25 | 中 | 第 2 批 | P0 |
| USA-01 | 用户能理解 X | 9 | 低 | 第 2 批 | P1 |
| ... | ... | ... | ... | ... | ... |

## 优先级分组

### P0 - 关键假设（立即验证）
| 假设ID | 描述 | 验证方式 | 计划时间 |
|--------|------|---------|---------|
| VAL-01 | ... | ... | ... |
| VAL-02 | ... | ... | ... |

### P1 - 重要假设（近期验证）
...

### P2 - 次要假设（稍后验证）
...

### P3 - 低优先级（可跳过）
...
```

### Step 6: 制定验证计划 (20-30 分钟)

**计划内容**:

#### 6.1 验证批次
```
第 1 批（本周）：
- VAL-01：用户有 X 需求
- MAR-01：市场规模足够大

第 2 批（下周）：
- VAL-02：用户愿意付费
- FEA-01：技术能实现

第 3 批（第 3 周）：
- USA-01：用户能理解产品
- VIA-01：能盈利
```

#### 6.2 验证时间表
```
## 验证时间表

| 批次 | 假设 | 验证方式 | 开始时间 | 结束时间 | 负责人 |
|------|------|---------|---------|---------|--------|
| 第 1 批 | VAL-01 | 用户访谈 | W1 周一 | W1 周五 | PM |
| 第 1 批 | MAR-01 | 数据调研 | W1 周一 | W1 周三 | PM |
| 第 2 批 | VAL-02 | 预售测试 | W2 周一 | W2 周五 | PM + 运营 |
| ... | ... | ... | ... | ... | ... |
```

#### 6.3 资源分配
```
## 资源分配

| 验证任务 | 所需资源 | 来源 | 预算 |
|---------|---------|------|------|
| 用户访谈 | 用户招募 10 人 | 用户池 | 1000 元 |
| 数据调研 | 行业报告 | 购买 | 500 元 |
| 预售测试 | 落地页 + 广告 | 设计 + 投放 | 5000 元 |
| ... | ... | ... | ... |
```

## 常见错误

- ❌ 只看重要性，忽视验证成本
- ❌ 忽视假设间的依赖关系
- ❌ 优先级过多，失去焦点
- ❌ 没有考虑资源限制
- ❌ 验证时间表过于乐观
