# 示例：使用 AI PM 创建一个社交 APP

本文档展示如何使用 AI PM 脚手架从零开始创建一个社交 APP。

## 场景

你想做一个「基于兴趣的线下活动匹配」社交 APP，但需求还很模糊。

## Step 1: 项目初始化

```bash
# 创建项目目录
mkdir activity-match-app
cd activity-match-app

# 初始化 AI PM
ai-pm create . --agents all
```

系统会生成：
- `.claude/skills/` - Claude Code 技能
- `.agents/skills/` - Codex/Cursor 技能
- `.ai-pm/` - 自进化引擎核心
- `CLAUDE.md` - 项目指令
- `PROJECT-PHASE.md` - 项目阶段状态

## Step 2: 战略规划

在 Claude Code 中说：

```
我想做一个「基于兴趣的线下活动匹配」社交 APP
```

系统自动路由到 `pm-center`，并推荐：

```
检测到新产品启动场景。推荐执行 /strategize 命令：

/strategize 将依次执行：
1. business-model-canvas - 梳理商业模式
2. swot-analysis - 分析优劣势
3. value-proposition - 明确价值主张
4. product-strategy - 制定产品战略

是否执行？
```

回复「是」后，系统开始执行：

### 2.2 SWOT 分析

基于商业模式，系统生成：

```
# SWOT 分析

## 优势 (Strengths)
- 差异化定位：聚焦线下活动匹配，避开线上社交红海
- 价值明确：解决「想参加活动但找不到合适的」痛点
- 变现路径清晰：会员 + 推广双模式

## 劣势 (Weaknesses)
- 冷启动难题：需要同时积累用户和活动
- 技术门槛：匹配算法需要大量数据训练
- 运营重：线下活动涉及安全保障

## 机会 (Opportunities)
- 后疫情时代线下社交需求爆发
- 现有产品（豆瓣、陌陌）功能不够聚焦
- Z 世代注重兴趣社交

## 威胁 (Threats)
- 大厂（微信、抖音）可能推出类似功能
- 用户可能只用一次就流失
- 活动质量难以控制
```

### 2.3 价值主张

系统使用 JTBD 框架引导：

```
# 价值主张 (JTBD)

## Who (目标用户)
25-35 岁城市白领，工作忙碌，想拓展社交圈，对现有社交方式不满意

## Why (动机)
- 现有社交 APP 太浮躁，找不到真诚的朋友
- 想参加活动但不知道哪里有合适的
- 一个人去活动尴尬，想找到同行者

## What before (现有方案的问题)
- 豆瓣同城：活动信息全但匹配不智能
- 陌陌：过于侧重约会，不够纯粹
- 微信群：信息分散，容易错过

## How (解决方案)
- 智能匹配：基于兴趣、时间、地点推荐活动
- 社交连接：找到想参加同一活动的人，可以一起报名
- 信任机制：用户评价、活动认证，降低参与门槛

## What after (预期效果)
- 发现感兴趣的活动：从「不知道有什么活动」到「每周都有想去的」
- 认识志同道合的朋友：从「一个人参加活动」到「认识了 10+ 同好」
- 拓展社交圈：从「同事和同学」到「兴趣社群」

## Alternatives (竞品)
- 豆瓣同城：活动更全，但匹配不智能
- Meetup：专注活动组织，社交属性弱
- 陌陌：偏约会，不够纯粹
```

### 2.4 产品战略

系统生成 9 段式产品战略：

```
# 产品战略

## 1. 愿景
成为城市白领的首选兴趣社交平台，让每个人都能找到志同道合的朋友。

## 2. 目标用户
核心用户：25-35 岁城市白领，一线城市，月收入 10k+，注重生活品质

## 3. 价值主张
智能匹配线下活动 + 社交连接

## 4. 权衡取舍
- 做深 vs 做广：先深耕一线城市，再扩展二三线
- 数量 vs 质量：活动质量优先，控制增长速度
- 免费 vs 付费：核心功能免费，增值服务收费

## 5. 成功指标
- 北极星指标：月度成功匹配数（用户参加活动并产生社交连接）
- 一级指标：MAU、7 日留存、活动参与率
- 二级指标：匹配准确率、用户满意度、付费转化率

## 6. 增长策略
- 产品驱动：极致匹配体验带来口碑传播
- 社交裂变：邀请好友一起参加活动获得奖励
- 合作增长：与活动平台、场地合作导流

## 7. 核心能力
- 智能匹配算法
- 活动数据库
- 社交关系链

## 8. 防御性
- 网络效应：用户越多，匹配越精准
- 数据壁垒：活动数据和用户行为数据
- 社区文化：形成独特的兴趣社区氛围

## 9. 里程碑
- Q1: MVP 上线，验证核心假设
- Q2: 达到 1 万 MAU，优化匹配算法
- Q3: 开启付费功能，验证商业模式
- Q4: 扩展到 5 个城市，MAU 达到 5 万
```

### 3.2 准备用户访谈

系统生成访谈脚本：

```
# 用户访谈脚本

## 访谈目标
验证假设：用户有「通过活动认识朋友」的需求

## 招募条件
- 25-35 岁城市白领
- 过去 3 个月参加过至少 1 次线下活动
- 有拓展社交圈的意愿

## 访谈问题

### 暖场（5 分钟）
1. 能简单介绍一下自己吗？
2. 平时有什么兴趣爱好？

### 核心探索（20 分钟）
3. 过去 3 个月参加过哪些活动？是怎么知道这些活动的？
4. 参加活动时有什么感受？有没有遇到想认识的人？
5. 参加活动后有没有保持联系的朋友？为什么能/不能保持联系？
6. 找活动时遇到的最大困难是什么？
7. 如果有一个 APP 能根据你的兴趣推荐活动，你觉得怎么样？
8. 如果能提前看到谁也想去同一个活动，你会想认识他们吗？

### 支付意愿（5 分钟）
9. 如果这个 APP 能帮你找到真正感兴趣的活动和志同道合的朋友，你愿意付费吗？
10. 你期望的功能是什么？愿意付多少钱？

### 结束
11. 还有什么想说的吗？
12. 如果我们做这个产品，你愿意成为种子用户吗？
```

### 3.3 设计验证实验

系统建议低成本验证方式：

```
# 验证实验设计

## 实验 1: 假门测试（2 周）
**假设**: 用户有「通过活动认识朋友」的需求
**方法**: 在小红书发布「城市活动推荐」内容，看用户是否点击
**成功标准**: 点击率 ≥ 5%，评论中有「想认识同好」的需求
**成本**: 内容制作 2 天

## 实验 2: 落地页测试（1 周）
**假设**: 用户愿意为精准匹配付费
**方法**: 制作产品介绍落地页，展示会员功能，看用户是否愿意预注册
**成功标准**: 预注册率 ≥ 3%，其中愿意付费的比例 ≥ 10%
**成本**: 落地页制作 1 天

## 实验 3: 数据源可行性（1 周）
**假设**: 能接入足够的活动数据源
**方法**: 调研豆瓣同城、Meetup 等平台的数据开放情况
**成功标准**: 至少找到 2 个可接入的数据源，日更新活动 ≥ 100 个
**成本**: 调研 3 天

## 实验执行顺序
Week 1: 实验 1 + 实验 3 并行
Week 2: 根据结果决定是否执行实验 2
```

### 4.1 PRD 示例

系统生成结构化 PRD：

```markdown
# 产品需求文档 - ActivityMatch v1.0

## 产品概述
ActivityMatch 是一个基于兴趣的线下活动匹配社交 APP。

## 目标用户
核心用户：25-35 岁城市白领，一线城市

## 核心功能

### P0 功能（MVP 必须有）
1. 用户注册登录（手机号 + 验证码）
2. 兴趣标签选择（从预设标签中选择 3-5 个）
3. 活动列表浏览（按时间、地点、兴趣筛选）
4. 活动详情页（时间、地点、费用、已报名人数）
5. 活动报名（一键报名 + 联系方式填写）
6. 我的已报名活动

### P1 功能（第一版要有）
1. 智能推荐（基于兴趣标签推荐活动）
2. 活动参与者列表（查看谁也报了名）
3. 私信功能（与参与者聊天）
4. 用户评价（活动后评价）

### P2 功能（后续迭代）
1. 活动组织者入驻
2. 会员体系
3. 活动推广
4. 社区动态

## 数据埋点
- 页面浏览：首页、活动详情、我的活动
- 按钮点击：报名、筛选、推荐、私信
- 业务事件：注册完成、活动报名成功、评价完成

## 验收标准
- 用户能在 3 步内完成注册并看到推荐活动
- 活动列表加载时间 < 2 秒
- 报名流程 < 30 秒
- 崩溃率 < 0.1%
```
