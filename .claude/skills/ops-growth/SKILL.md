# 运营与商业化中心（Ops & Growth）

## 目的

让产品活下去、用户增长、赚到钱。本 Skill 集合了 7 个角色的集体工作方法——产品上线只是开始，持续运营和商业化才是 APP 的生命线。

**核心原则**: 好产品 + 好运营 = 增长飞轮。只做产品不做运营 = 建了高速公路没人开车。

## 角色全景

| 角色 | 一句话职责 | 主要产出 |
|------|-----------|----------|
| **用户运营** | 拉新、促活、留存——让用户来了就不走 | 拉新活动方案、用户分层策略、Push 推送计划、召回方案 |
| **内容运营** | 管好 APP 里的内容生态 | 审核规则、创作者激励方案、话题活动、内容分发策略 |
| **渠道投放/增长** | 花钱买用户——最低成本获取高质量用户 | ASO 优化方案、投放计划、素材 AB 测试、ROI 分析报告 |
| **商业化/广告运营** | 流量变现——把 DAU 变成收入 | 广告位策略、会员方案、电商活动、ARPU 提升方案 |
| **法务合规** | 保证 APP 不违法、不被告、不下架 | 用户协议、隐私政策、IP 审核、监管应对 |
| **内容审核** | 不让违规内容出现在 APP 里 | 审核规则库、AI+人工审核流程、举报处理 |
| **客服** | 用户遇到问题的第一接触点 | FAQ/知识库、工单处理、用户反馈汇总 |

## 触发条件

以下场景调用本 Skill：
- APP 已上线或即将上线，需要运营支持
- 用户增长或留存出现下滑
- 商业化策略需要调整或优化
- 应用商店 ASO 优化
- 付费投放计划制定
- 法务合规审查（用户协议更新、隐私政策修改）
- 客服反馈中出现高频问题需要产品改进
- 内容审核标准和流程建立

## 核心工作流程

### Phase A: 运营规划
1. 对齐产品路线图（来自 pm-center）和业务目标
2. 拆解运营 KPI：拉新量、留存率、活跃度、收入
3. 制定运营日历：关键时间节点、活动节奏、资源需求
4. 输出运营方案

### Phase B: 用户增长
1. **ASO 优化**: 标题/副标题/关键词/描述优化，AB 测试图标和截图
2. **付费投放**: 搭建投放计划（平台选择、人群定向、出价策略、预算分配）、素材 AB 测试
3. **裂变活动**: 邀请有礼、拼团、助力——优化每一步转化率
4. 监控 CPI/ROI/LTV，动态调整预算分配

### Phase C: 用户运营与留存
1. 用户分层（RFM 模型：高价值/中价值/低价值/流失）
2. 差异化策略：高价值→VIP权益，低价值→优惠券激活，流失→短信/推送召回
3. Push 推送策略：时机/内容/频次/人群
4. 签到/任务/积分体系设计
5. 社群运营维护

### Phase D: 商业化运营
1. **广告**: 管理广告位（位置/样式/频次/底价），优化填充率和 eCPM
2. **会员**: 权益设计、定价策略、拉新活动、续费召回
3. **电商/带货**: 大促活动、商家激励、商品推荐策略
4. 监控 ARPU/ARPPU/付费率

### Phase E: 内容与社区运营
1. 制定内容审核规则（政治/色情/暴力/违法/侵权/广告/低质）
2. 搭建 AI 审核 + 人工审核双保险流程
3. 创作者激励体系（流量扶持/分成/认证）
4. 话题/活动策划，提升内容供给和用户参与度

### Phase F: 合规与法务
1. 起草/审查用户服务协议和隐私政策（国内版 + 海外版）
2. 数据合规检查：个人信息采集最小化、跨境传输合规
3. 知识产权：商标、第三方素材版权审核
4. 营销内容合规审查（「最」「第一」等禁用词）
5. 应用商店审核准备

### Phase G: 客服与反馈闭环
1. 维护 FAQ/帮助中心/智能客服机器人
2. 处理用户工单（账号/支付/退款/功能咨询）
3. 收集高频问题和 Bug 反馈 → 同步给 pm-center
4. 重大投诉升级机制

## 工具栈

- 推送: 个推 / 极光 / Firebase Cloud Messaging
- ASO: 七麦数据 / App Annie / Sensor Tower
- 广告投放: 巨量引擎 / 广点通 / Google Ads / Apple Search Ads
- 归因: AppsFlyer / Adjust
- CRM/SCRM: 企业微信 / 内部用户运营平台
- 客服: 网易七鱼 / 智齿 / Zendesk
- 内容审核: 内部审核系统 + AI 模型
- 法务: 北大法宝 / 监管法规数据库

## 输出要求

本 Skill 执行后（按需）必须产出：
1. **运营方案** — 含 KPI、活动日历、资源需求
2. **投放计划** — 含渠道/预算/素材/ROI 预期
3. **用户分层策略** — 含差异化运营方案
4. **商业化方案** — 广告/会员/电商
5. **合规检查清单** — 用户协议/隐私政策/内容审核规则
6. **客服 FAQ/知识库**

## 验收标准

- [ ] 运营 KPI 明确且可度量
- [ ] 合规检查在发布前完成
- [ ] 客服响应时间在 SLA 以内
- [ ] 法务审核完成（所有公开内容）
- [ ] 内容审核流程运转正常（延迟在可接受范围）
- [ ] 投放 ROI 正向或处于可接受的测试阶段

## 禁止事项

- ❌ 在法务审核通过前发布用户协议/隐私政策
- ❌ 广告过度影响用户体验（弹窗/诱导点击）
- ❌ 用户数据未经授权用于投放/商业化
- ❌ 客服承诺产品做不到的功能
- ❌ 内容审核规则留有明显漏洞
- ❌ 运营活动未评估风险（羊毛党/服务器承压）

## 上游交付物评分（接收上游交付物时执行）

本 Skill 接收来自 **pm-center**（功能说明+埋点）和 **dev-factory**（配置文档）的双上游输入：

### 对 pm-center 的功能说明评分

| 维度 | 评什么 |
|------|--------|
| **完整性** | 功能说明是否足够运营理解产品？埋点事件是否定义清晰？ |
| **清晰度** | 产品定位和目标用户描述是否明确？运营能否据此设计活动？ |
| **及时性** | 功能说明是否在运营方案制定前到位？ |
| **可用性** | 运营能否直接按功能说明策划活动？ |

### 对 dev-factory 的配置文档评分

| 维度 | 评什么 |
|------|--------|
| **完整性** | 后台开关/灰度配置/运营手册是否齐全？ |
| **清晰度** | 配置项是否都有说明？运营能否独立操作后台？ |
| **及时性** | 配置文档是否在上线前到位？ |
| **可用性** | 运营能否直接按文档操作，不需开发陪同？ |

**打分流程**: 接收上下游交付物 → 对照 `.claude/SCORING-RUBRIC.md` 分别打分 → 任一方总分 < 7 调用 `evolution-engine`
