# 研发中心（Dev Factory）

## 目的

把设计图纸和产品需求写成可运行的代码。APP 的骨骼和血肉。本 Skill 覆盖 15 个技术角色，是人数最多、分工最细的阶段。

**核心原则**: 一次只做一个 Task。不偏移到相邻工作。动力有用；伪装成生产力的范围蔓延不行。

## 角色全景

### 一、客户端/前端开发（APP 本体核心）

| 角色 | 技术栈 | 核心职责 |
|------|--------|----------|
| **iOS 工程师** | Swift / SwiftUI / OC / UIKit | iPhone/iPad APP 开发，App Store 上架，苹果生态适配（灵动岛/深色模式/IAP） |
| **Android 工程师** | Kotlin / Java / Jetpack Compose | 安卓 APP 开发，厂商碎片化适配（小米/华为/OPPO），Google Play 上架 |
| **跨端工程师** | Flutter (Dart) / React Native (TS) | 一套代码双端产出，降本增效，处理原生 Bridge 调用 |
| **鸿蒙工程师** | ArkTS / ArkUI / DevEco Studio | HarmonyOS APP 开发，原子化服务，多设备流转 |

## 触发条件

以下场景调用本 Skill：
- PRD + 设计稿已就绪，开发可以启动
- 需要技术方案设计和架构评审
- 新功能需要编码实现
- Bug 修复或性能优化
- CI/CD 流水线搭建或维护
- 构建打包和发布准备
- 技术选型或架构决策

## 核心工作流程

### Phase A: 技术方案设计
1. 客户端架构师 + 服务端架构师 联合评审
2. 读取 `PRD.md`（来自 pm-center）+ 设计稿（来自 design-studio）
3. 输出技术方案：
   - 架构选型（MVVM/MVI/VIPER、微服务拆分边界）
   - 接口定义（RESTful/gRPC 路径、参数、返回值、错误码）
   - 数据库变更（新增表/字段、索引、迁移方案）
   - 关键流程图和时序图
   - 风险点和回滚方案
4. 技术方案评审会（TL + 开发 + 测试参加）
5. 通过后进入开发

### Phase B: 后端先行开发
1. 后端工程师创建接口骨架和 Mock 数据
2. DBA 审核并执行数据库变更
3. 输出接口文档（Swagger/YAPI/内部平台）
4. **原则**: 接口文档和 Mock 先行，让客户端不被后端阻塞

### Phase C: 客户端开发
1. iOS/Android/跨端 工程师并行开发
2. 对照设计稿实现 UI（像素对齐）
3. 先用 Mock 数据跑通流程，后端就绪后切换
4. 实现所有交互状态（加载中/空数据/错误/异常）
5. 特殊处理：弱网、低电量、中断（来电/切后台）

### Phase D: 联调与集成
1. 客户端切换后端真实接口
2. 端到端流程验证
3. 异常分支测试
4. 算法/音视频等能力模块集成

### Phase E: 基础设施与性能
1. DevOps 维护 CI/CD 流水线（提交→构建→单测→打包）
2. 编译/打包工程师优化构建速度和出包流程
3. 客户端架构师做性能专项：启动速度、内存、包体积、卡顿率
4. 安全工程师做安全扫描和加固

### Phase F: 提测前自检
1. 逐条过自检 Checklist
2. P0 核心流程自测通过（登录→浏览→下单→支付 等）
3. 无 P0 级别崩溃或阻塞 Bug
4. 输出自检报告 → 交付 qa-gate

## 实现规则

- 遵循项目已有的代码结构和风格
- 在发明新抽象之前，先使用已有的库和辅助函数
- 仅在能消除真实复杂度时才添加抽象
- UI 优先对齐设计稿，其次是设计规范，再次是 PRD
- 不要悄悄修改不相关的文件
- **一次只做一个 Task**，不允许把多个 Task 混在一个大改动里
- 编译不过不能进入下一个 Task

## 提测前自检清单

每个 Task 完成前必须确认：
- [ ] 功能行为与 PRD 验收标准一致
- [ ] UI 实现与设计稿对齐（颜色/间距/字号/状态）
- [ ] 所有交互状态已实现（默认/加载中/空数据/错误/禁用）
- [ ] 无崩溃（正常流程 + 异常流程）
- [ ] 接口请求有 loading 和 error 处理
- [ ] 敏感数据不打印到日志
- [ ] 代码已通过 code review（如项目已启用）

## 输出要求

本 Skill 执行后必须产出：
1. **可运行的构建包**（IPA/AAB/APK 或可部署的服务端容器）
2. **接口文档**（所有新增/修改的 API）
3. **数据库变更记录**（迁移脚本）
4. **提测自检报告**
5. **技术方案文档**（如涉及架构变更）

## 验收标准

- [ ] 构建编译通过，可安装运行
- [ ] 所有 P0 验收标准（来自 PRD）满足
- [ ] 自检中无 P0/P1 阻塞 Bug
- [ ] 性能基线和上一版本对比无劣化
- [ ] UI 实现与设计稿偏差在可接受范围内
- [ ] 关键路径有错误处理和用户提示

## 禁止事项

- ❌ PRD 和设计稿缺失时强行写代码
- ❌ 一个 Task 跨多个不相关模块修改
- ❌ 编译不过就提测
- ❌ 跳过自检直接扔给 QA
- ❌ 修 A 功能顺带改 B 功能（范围蔓延）
- ❌ 忽略异常状态处理（只写 happy path）
- ❌ 调试代码/硬编码密钥/测试日志带入生产包
