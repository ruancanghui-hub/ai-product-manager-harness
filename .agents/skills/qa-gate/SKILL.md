# 测试中心（QA Gate）

## 目的

保证 APP 不出 bug，用户体验不翻车。本 Skill 集合了 6 个测试角色的集体工作方法——从功能测试到合规审查，每一关都是上线前的必要防线。

**核心原则**: 质量门不放水。P0 bug 不清零不发布。

## 角色全景

| 角色 | 一句话职责 | 主要产出 |
|------|-----------|----------|
| **功能测试** | APP 上线前最后把关人，亲手测所有功能 | 测试用例、Bug 报告、回归验证记录 |
| **自动化测试** | 写脚本替代人工重复测试 | UI 自动化脚本、接口自动化脚本、CI 集成 |
| **性能测试** | 保证不卡顿、不耗电、启动快 | 启动/内存/功耗/流量/服务端压测报告 |
| **兼容性测试** | 保证几百种机型上都能正常用 | 机型兼容性矩阵报告、云真机截图对比 |
| **安全测试** | 模拟攻击找漏洞，防数据泄露 | 渗透测试报告、代码安全扫描结果 |
| **合规测试** | 保证不被应用商店下架 | 审核清单检查结果、隐私合规报告 |

## 触发条件

以下场景调用本 Skill：
- 开发提测（收到来自 dev-factory 的构建包）
- 新功能或新版本需要全量 QA
- 怀疑性能回归
- 应用商店提交前的合规审查
- Bug 修复后的回归验证
- 用户反馈的 bug 需要复现和定位

## 核心工作流程

### Phase A: 测试计划
1. 读取 `PRD.md`（来自 pm-center），理解测试范围
2. 确定测试策略：哪些手工测、哪些自动化、哪些专项测
3. 评估测试工作量和时间
4. 输出测试计划：范围/策略/排期/人员/通过标准

### Phase B: 用例设计与评审
1. 根据 PRD + 设计稿 编写测试用例：
   - 正常流程 × N
   - 异常流程（错误参数/超时/并发）
   - 边界值（空字符串/超长文本/0/负数/最大值）
   - 权限场景（未登录/已登录/已过期/权限不足）
   - 中断场景（来电/通知/切换后台/低电量）
2. 用例评审：PM + 开发 + 测试一起过，确保覆盖无遗漏

### Phase C: 功能测试执行
1. 搭建测试环境（服务器/账号/数据）
2. 逐条执行测试用例，标记通过/失败
3. 发现 Bug → 提交：
   - 标题（简洁描述）
   - 优先级（P0 崩溃/阻塞 | P1 核心流程 | P2 体验/边缘）
   - 复现步骤（Step 1 → 2 → 3）
   - 预期结果 vs 实际结果
   - 截图/录屏 + 日志
   - 机型/系统版本
4. 每日和开发同步 Bug 状态
5. P0 级别 Bug 要求 24 小时内修复

### Phase D: 自动化测试
1. 写 UI 自动化脚本覆盖核心回归流程
2. 写接口自动化脚本覆盖所有接口（正常 + 异常参数）
3. 集成到 CI 流水线（每次提交/每天凌晨自动跑）
4. 维护不稳定用例，降低误报率

### Phase E: 专项测试
1. **性能测试**: 启动耗时（冷/热）、页面 FPS、内存峰值/泄漏、功耗、首次安装流量
2. **兼容性测试**: TOP 50 机型遍历核心流程、云真机截图对比、厂商 ROM 特殊行为
3. **安全测试**: 抓包/篡改/重放/SQL注入/XSS、逆向检查、隐私数据检查
4. **合规测试**: App Store 审核清单逐条过、隐私弹窗/ATT/数据删除检查

### Phase F: 质量门
1. Bug 收敛分析：新增速率下降、关闭速率上升
2. P0/P1 Bug 必须清零
3. P2 Bug 有明确处理计划（修复或记录）
4. 性能/兼容/安全/合规无阻塞项
5. 输出质量签批结果：**通过（可发布）/ 驳回（需修复后重测）**

## Bug 优先级定义

| 级别 | 定义 | 处理时效 |
|------|------|----------|
| **P0** | 崩溃、白屏、数据丢失、支付失败、安全漏洞 | 24 小时内修复 |
| **P1** | 核心流程中断、功能与 PRD 严重不符、UI 严重错乱 | 当前迭代内修复 |
| **P2** | 体验问题、边缘场景异常、UI 轻微偏差 | 本迭代或下迭代修复 |

## 输入/输出契约

| 方向 | 对象 | 交付物 | 格式 |
|------|------|--------|------|
| **输入来自** | dev-factory | 构建包 + 自检报告 + 已知问题清单 | IPA/AAB/APK + 文档 |
| **输入来自** | pm-center | PRD（验收标准） | Markdown 文档 |
| **输入来自** | design-studio | 设计稿（视觉参照） | Figma/标注 |
| **输出给** | dev-factory | Bug 报告、回归结论、性能发现 | Bug 追踪系统 |
| **输出给** | pm-center | 质量报告（度量产品健康度） | 测试报告 |
| **输出给** | chief-orchestrator | 质量门结果（通过/驳回+原因） | 签批文件 |
| **输出给** | ops-growth | 已知问题清单（供客服参考） | 文档 |

## 工具栈

- Bug 管理: Jira / TAPD / Bugly
- 抓包: Charles / Wireshark / Proxyman / Burp Suite
- 自动化: XCTest / Espresso / Appium / pytest
- 性能: Instruments / Android Profiler / JMeter / Locust
- 兼容: 阿里云测 / WeTest / Firebase Test Lab
- 安全扫描: OWASP ZAP / Fortify / 自研平台

## 输出要求

本 Skill 执行后必须产出：
1. **测试计划** — 范围/策略/排期/通过标准
2. **测试用例清单** — 含覆盖范围说明
3. **Bug 清单** — 含优先级/状态/负责人
4. **测试执行报告** — 通过率/失败详情/截图/日志
5. **专项测试报告** — 性能基准对比/兼容矩阵/安全扫描/合规清单
6. **质量签批结果** — 通过 或 驳回（含阻塞项清单）

## 验收标准

- [ ] 所有 P0/P1 Bug 已关闭
- [ ] 核心流程测试用例 100% 通过
- [ ] 性能基准与上一版本对比无劣化
- [ ] TOP 50 机型兼容性通过
- [ ] 安全扫描无高危漏洞
- [ ] 合规清单全部通过
- [ ] 质量签批结果已同步给 chief-orchestrator

## 禁止事项

- ❌ 提测包冒烟不通过还继续测
- ❌ P0 bug 不追开发、默默标记「已知」
- ❌ 只测正常流程不测异常和边界
- ❌ 性能劣化不上报
- ❌ 合规不通过放行（「应该不会被拒吧」心态）
- ❌ 测试用例不评审就执行（可能漏关键场景）

## 上游交付物评分（接收上游交付物时执行）

本 Skill 接收来自 **pm-center**（验收标准）和 **dev-factory**（构建包+自检报告）的双上游输入：

### 对 pm-center 的验收标准评分

| 维度 | 评什么 |
|------|--------|
| **完整性** | 验收标准是否覆盖正常+异常+边界场景？是否每个 P0/P1 都有对应标准？ |
| **清晰度** | 验收标准是否可测试？「体验好」这种不可测标准是否被具体化？ |
| **及时性** | 验收标准是否在测试计划编写前到位？ |
| **可用性** | 测试能否直接按验收标准写用例？还是需要自己揣摩需求？ |

### 对 dev-factory 的提测质量评分

| 维度 | 评什么 |
|------|--------|
| **完整性** | 构建包是否可安装？自检报告是否诚实？已知问题清单是否列出？ |
| **清晰度** | 自检报告和已知问题描述是否清晰可理解？ |
| **及时性** | 是否按提测日交付？延期是否提前通知？ |
| **可用性** | 构建包能否直接进入测试？冒烟是否通过？ |

**打分流程**: 接收上下游交付物 → 对照 `.Codex/SCORING-RUBRIC.md` 分别打分 → 任一方总分 < 7 调用 `evolution-engine`
