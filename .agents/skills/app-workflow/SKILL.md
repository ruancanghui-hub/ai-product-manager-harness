# 全链路流水线（App Workflow）

## 目的

串联全部 7 个角色 Skill，形成一条从「想法」到「上线运营」的完整 APP 开发流水线。本 Skill 是**总指挥**——知道什么阶段该调用哪个 Skill、该产出什么、该满足什么质量门才能进入下一阶段。

**核心原则**: 不跳阶段。跳过的每一步，都会在后续以返工的形式加倍偿还。

## 触发条件

以下场景调用本 Skill：
- 用户说「我想做一个 XX APP」「开始一个新项目」「从零开始」
- 用户明确要求走全流程
- 当前项目未初始化，需要从头搭建
- 用户说「继续推进项目」

## 全生命周期流水线（9 个阶段）

```
Phase 0: 项目立项 ──→ Phase 1: 产品定义 ──→ Phase 2: 设计
                                                      ↓
Phase 8: 复盘迭代 ←── Phase 7: 运营 ←── Phase 6: 发布 ←── Phase 5: 测试
                                                      ↓
                              Phase 3: 技术方案 ──→ Phase 4: 开发
```

---

## Phase 0: 项目立项

**调用**: chief-orchestrator + mgmt-office

**目标**: 确定要不要做、谁来做、什么时候交付。

**输入**: 产品想法或业务需求

**流程**:
1. mgmt-office: 评估战略价值和资源可行性
2. chief-orchestrator: 建立项目框架（团队、排期、沟通机制）
3. 输出项目章程：目标、范围、团队、关键里程碑

**产出**:
- 项目章程（目标/范围/团队/里程碑）
- 第一次迭代计划（粗粒度）
- 团队通讯录和沟通渠道

**质量门**: ✅ 目标和范围明确，团队到位，关键里程碑已确定

---

## Phase 1: 产品定义

**调用**: pm-center（主导）

**目标**: 把想法变成结构化的产品需求。

**输入**: 项目章程、产品想法

**流程**:
1. pm-center Phase A-C: 需求澄清 → 用户研究（按需）→ PRD 编写
2. pm-center Phase D: 数据指标和埋点设计
3. pm-center Phase E: 需求评审会（召集设计+开发+测试）
4. pm-center Phase F: 排期建议给 chief-orchestrator

**产出**:
- `PRD.md`（结构化产品需求文档）
- 埋点事件文档
- 用研报告（如触发）
- 数据看板定义

**质量门**: ✅ PRD 评审通过，开发确认无模糊，每个 P0/P1 需求有可测试的验收标准

---

## Phase 2: 设计

**调用**: design-studio（主导）

**目标**: 把 PRD 变成可落地的界面设计。

**输入**: `PRD.md`

**流程**:
1. design-studio Phase A: 信息架构设计
2. design-studio Phase B: 交互设计（含所有状态）
3. design-studio Phase C: 视觉设计 + 设计系统
4. design-studio Phase D: 动效设计
5. design-studio Phase E: 多端适配
6. design-studio Phase F: 设计评审会

**产出**:
- 信息架构图
- 交互原型（含完整状态标注）
- 视觉设计稿（含深色模式）
- 设计系统和组件库
- 动效规格

**质量门**: ✅ 设计评审通过，所有状态已覆盖，开发确认可实现

---

## Phase 3: 技术方案

**调用**: dev-factory（架构师角色）

**目标**: 在写代码前确定技术路线和关键设计。

**输入**: `PRD.md` + 设计稿

**流程**:
1. dev-factory Phase A: 客户端架构师 + 服务端架构师联合设计
2. 输出技术方案：架构选型、接口定义、数据库变更、风险点
3. 技术方案评审会（TL + 开发 + 测试参加）

**产出**:
- 技术方案文档（架构/接口/DB/风险/回滚）
- 接口文档（初版，含 Mock）
- 数据库变更方案

**质量门**: ✅ 技术方案评审通过，接口定义和 DB 变更已被各端确认

---

## Phase 4: 开发

**调用**: dev-factory（全量角色）

**目标**: 把设计和技术方案写成可运行代码。

**输入**: PRD + 设计稿 + 技术方案

**流程**:
1. dev-factory Phase B: 后端先行（API + DB + Mock）
2. dev-factory Phase C: 客户端并行开发（先用 Mock 数据）
3. dev-factory Phase D: 联调与集成
4. dev-factory Phase E: 基础设施与性能优化
5. dev-factory Phase F: 提测前自检

**产出**:
- 可运行构建包（IPA/AAB/APK）
- 接口文档（终版）
- 数据库迁移脚本
- 提测自检报告
- CI/CD 流水线配置

**质量门**: ✅ 自检 Checklist 通过，P0 用例无阻塞 Bug，编译通过

---

## Phase 5: 测试

**调用**: qa-gate（主导）

**目标**: 全面验证质量，不放一个有 bug 的版本上线。

**输入**: 构建包 + 自检报告 + PRD 验收标准

**流程**:
1. qa-gate Phase A-B: 测试计划 + 用例设计评审
2. qa-gate Phase C: 功能测试执行 + Bug 追踪
3. qa-gate Phase D: 自动化测试
4. qa-gate Phase E: 专项测试（性能/兼容/安全/合规）
5. qa-gate Phase F: 质量门——P0/P1 清零检查

**产出**:
- 测试计划 + 用例
- Bug 清单
- 测试执行报告（通过率/失败详情）
- 专项测试报告（性能/兼容/安全/合规）
- 质量签批结果

**质量门**: ✅ P0/P1 Bug 清零，性能无劣化，兼容性通过，安全合规通过

---

## Phase 6: 发布

**调用**: dev-factory（DevOps 角色）+ chief-orchestrator

**目标**: 安全、可控地把新版本送到用户手中。

**输入**: 质量签批通过 + 构建包

**流程**:
1. chief-orchestrator: 发布前 Checklist 最后一轮确认
2. DevOps: 执行灰度发布（1% → 5% → 50% → 100%）
3. 每步观察：崩溃率、接口成功率、核心业务指标
4. 异常时执行回滚方案
5. 全量后通知运营团队同步上线活动

**产出**:
- 灰度发布记录和数据
- 发布说明
- 已知问题和回滚方案

**质量门**: ✅ 灰度期间无 P0 告警，全量后 24h 核心指标稳定

---

## Phase 7: 运营

**调用**: ops-growth（主导）+ pm-center（数据分析）

**目标**: 让产品被用户使用、喜欢、付费。

**输入**: 已上线版本 + 产品目标 + 埋点数据

**流程**:
1. ops-growth Phase B: 用户增长（ASO/投放/裂变）
2. ops-growth Phase C: 用户运营与留存
3. ops-growth Phase D: 商业化运营
4. ops-growth Phase E: 内容与社区运营
5. ops-growth Phase G: 客服与反馈收集
6. pm-center Phase F: 数据回收和版本复盘

**产出**:
- 运营活动方案和执行记录
- 投放计划和 ROI 报告
- 用户反馈汇总
- 版本数据复盘报告

**质量门**: ✅ 核心指标已回收，用户反馈已汇总，客服体系运转

---

## Phase 8: 复盘迭代

**调用**: chief-orchestrator + pm-center

**目标**: 从这个版本中学习，规划下个版本。

**输入**: 版本数据 + 用户反馈 + 运营结果

**流程**:
1. chief-orchestrator: 版本复盘——延期原因/瓶颈分析/改进建议
2. pm-center: 提炼下版本需求
3. mgmt-office: 更新 OKR 和战略方向（如需要）
4. 输出下版本计划 → 回到 Phase 1

**产出**:
- 版本复盘报告（延期原因/瓶颈/改进项）
- 下版本需求池（初版）
- 团队效能趋势

**质量门**: ✅ 复盘完成，改进项有负责人和截止日期，下版本需求已入池

---

## 阶段转移规则

1. **每个阶段完成时**，chief-orchestrator 执行质量门检查
2. **质量门通过** → 进入下一阶段
3. **质量门不通过** → 当前阶段返工，直到通过
4. **用户要求跳过某阶段** → 必须明确告知风险，记录在案，允许继续
5. **阶段间的 Skill 调用** → 每个 Skill 启动时检查输入是否满足最低要求
6. **每个阶段结束时** → 下游对上游交付物打分（参照 `.Codex/SCORING-RUBRIC.md`），不及格触发 `evolution-engine`

## 自进化打分节点（嵌入流水线）

在每个阶段转换前，下游 Skill 对上游 Skill 打分：

| 阶段转换 | 打分方 | 被打分方 | 评什么 |
|----------|--------|----------|--------|
| Phase 1→2 | design-studio | pm-center | PRD 质量 |
| Phase 2→3 | dev-factory | design-studio | 设计交付质量 |
| Phase 3→4 | dev-factory | pm-center | PRD 技术可执行性 |
| Phase 4→5 | qa-gate | dev-factory | 提测质量 |
| Phase 5→6 | qa-gate | pm-center | 验收标准质量 |
| Phase 6→7 | ops-growth | pm-center + dev-factory | 功能说明 + 配置文档 |
| Phase 7→8 | chief-orchestrator | 所有团队 | 信息透明度 |

**不及格处理**: 任一打分 < 7 分 → 暂停流水线 → 调用 `evolution-engine` → 上游 Skill 改进 → 重新交付 → 重新打分 → 通过后继续

## 技能互调快速指引

当用户在任意阶段提出跨阶段需求时：

| 用户说 | 调到哪个 Skill | 说明 |
|--------|---------------|------|
| "需求变了" | pm-center | 回到 Phase 1，更新 PRD |
| "设计要调整" | design-studio | 回到 Phase 2，同步更新下游 |
| "这个技术方案行不通" | dev-factory | 回到 Phase 3，重做技术方案 |
| "这里有个 bug" | qa-gate | 进入测试流程（如在 Phase 7 则是 bug-fix 流程） |
| "上线后数据不好" | pm-center + ops-growth | 同时分析数据和调整运营 |
| "团队要调整" | mgmt-office | 管理决策流程 |
| "卡住了，不知道谁在等谁" | chief-orchestrator | 总调度介入，梳理依赖 |

## 持续推进模式

当用户说「继续」「接着做」「推进」时：
1. 读取 `PROJECT-PHASE.md`（或等效状态文件）
2. 确认当前阶段和已通过的质量门
3. 如果当前阶段未完成 → 继续该阶段的下一步
4. 如果当前阶段已完成 → 执行质量门 → 进入下一阶段
5. 如果全部阶段完成 → 进入 Phase 8 复盘迭代

## 输出要求

本 Skill 串联执行过程中，所有中间产出物由各阶段 Skill 负责，本 Skill 负责：
1. **PROJECT-PHASE.md** — 当前阶段、已通过质量门、下一步行动
2. **全链路状态跟踪** — 每个阶段的输入/输出/质量门/状态

## 验收标准

- [ ] 每个阶段产出了必需交付物才进入下一阶段
- [ ] 每个质量门都有明确的通过/驳回记录
- [ ] 跨阶段跳转（如开发中改需求）有明确的回溯流程
- [ ] 用户可以随时说「继续」，系统知道当前该做什么
- [ ] 每个阶段转换前的上下游打分已完成并记录
