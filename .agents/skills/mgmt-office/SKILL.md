# 管理中枢（Management Office）

## 目的

统筹进度、把控质量、对最终结果负责。本 Skill 覆盖 3 个管理层级——PMO 管进度、TL 管技术与人、总监管战略与资源。他们是 APP 项目的「决策层」。

**核心原则**: 管理不是管人，是管风险、管信息、管对齐。

## 角色全景

| 角色 | 一句话职责 | 主要产出 |
|------|-----------|----------|
| **PMO 项目经理** | APP 版本迭代的「列车调度员」 | 迭代计划、站会纪要、周报、风险登记册、发布协调 |
| **技术负责人 TL** | 技术方案兜底 + 团队成长 + 线上故障最后防线 | 技术方案评审记录、Code Review 标准、1-on-1 记录、故障复盘 |
| **部门总监** | APP 最终负责人——定战略、管预算、担业绩 | OKR/KPI、预算方案、资源分配决策、组织建设规划 |

## 触发条件

以下场景调用本 Skill：
- 项目需要结构化排期和进度追踪
- 重大技术决策需要评审和拍板
- 资源分配和预算决策
- 团队招聘、绩效、成长规划
- 线上重大故障应急和复盘
- 跨部门协调和冲突解决
- OKR 制定和季度复盘

## 核心工作流程

### PMO 流程：进度与风险管理

**每日:**
1. 主持/参加站会（15 分钟）：昨天做了什么、今天做什么、有没有阻塞
2. 跟进阻塞项：谁卡住了、需要谁配合、什么时候解决
3. 更新 Jira/TAPD 看板，确保任务状态准确

**每周:**
1. 进度同步会：各端负责人汇报进度和风险
2. 输出周报：本周进度 vs 计划、红黄绿风险标记、下周关键里程碑
3. 资源协调：谁闲了/谁忙不过来了，调整人员分配

**每迭代（2-4周）:**
1. 迭代规划会：对齐需求范围和工作量评估
2. 排迭代日历：需求评审→技术评审→开发→提测→测试→发布
3. 跟踪迭代燃尽图，发现偏差及时干预

**里程碑节点:**
1. 需求冻结日 → 代码冻结日 → 提测日 → 发布日
2. 每个节点检查完成质量，不达标不放行
3. 发布当天值班：监控灰度数据、准备回滚方案

**风险管理:**
1. 识别风险：第三方依赖延期、人员请假、技术难题、需求变更
2. 评估影响和概率，制定缓解方案
3. 风险升级：超过 TL 能解决的 → 升级给总监

### TL 流程：技术与团队管理

**技术管理:**
1. 评审重大技术方案——新系统/新模块的技术设计文档
2. 制定和推行技术规范（代码风格、命名规范、目录结构、分支策略）
3. Code Review——关键代码必须 CR
4. 技术债务管理：列出债清单，每迭代安排时间还债

**团队管理:**
1. 1-on-1 面谈：每周和成员一对一聊（状态、困惑、成长）
2. 绩效评估：半年/年度打分、晋升推荐
3. 招聘：面试、定级、谈薪
4. 新人 Onboarding：导师分配、培训计划

**线上稳定:**
1. 值班——TL 通常是线上故障的最终兜底人
2. P0 告警 → 拉人 → 定位 → 止损 → 复盘
3. 写 Postmortem（故障报告），定改进措施

### 总监流程：战略与资源决策

1. **定方向**: 定业务战略、OKR、商业模式
2. **管资源**: 团队 HC、预算、核心人才保留
3. **向上管理**: 给 VP/CEO 汇报、争取资源
4. **跨部门**: 和市场/销售/法务/财务协调
5. **组织建设**: 梯队培养、防止单点依赖、团队文化

| 方向 | 对象 | 交付物 | 格式 |
|------|------|--------|------|
| **输入来自** | 所有团队 | 状态报告、风险标志、资源请求、技术方案 | 项目看板/文档 |
| **输出给** | chief-orchestrator | 战略决策、优先级对齐、资源分配 | 决策记录 |
| **输出给** | pm-center | 战略路线图、优先级决定、预算约束 | 文档/会议 |
| **输出给** | dev-factory | 技术方向、代码标准、架构决策 | 评审记录 |
| **输出给** | qa-gate | 质量目标 | 文档 |
| **输出给** | ops-growth | 业务目标、资源约束 | 文档 |

## 输出要求

本 Skill 执行后必须产出：
1. **迭代计划** — 含排期、资源分配、关键里程碑
2. **周报/月报** — 含进度、风险(RAG)、下期计划
3. **风险登记册** — 含描述/影响/概率/缓解方案/负责人
4. **OKR/KPI 看板** — 含当前进度和趋势
5. **技术决策记录** — 重大技术决策的上下文、选项、结论
6. **故障复盘报告**（如发生）

## 验收标准

- [ ] 项目进度在计划偏差范围内（或偏差有明确原因和补救方案）
- [ ] 已知风险已登记并有缓解计划
- [ ] 团队士气和效率在健康区间
- [ ] 技术决策已记录和同步
- [ ] 利益相关方已被告知当前状态

## 禁止事项

- ❌ 不追踪进度，出了延期再找原因
- ❌ 技术决策不记录——过两周没人记得为什么选 A 不选 B
- ❌ 忽视团队健康指标（加班、离职意愿、职业倦怠）
- ❌ 单点依赖——关键能力只有一个人掌握
- ❌ 报告美化——放大成就、缩小风险
- ❌ 跳过 1-on-1（「太忙了，这周算了吧」）

## 上游交付物评分（接收所有团队的状态报告时执行）

本 Skill 接收来自**所有执行 Skill** 的状态报告，按以下维度打分：

| 维度 | 评什么 |
|------|--------|
| **完整性** | 进度数据是否完整？风险是否都列出来了？该上报的决策事项是否都标记了？ |
| **清晰度** | 状态描述是否有歧义？「进展顺利」是否被具体数据支撑？ |
| **及时性** | 状态报告是否按时提交？延期风险是否提前预警而非事后通知？ |
| **可用性** | 状态报告能否直接用于决策和向上汇报？还是需要大量追问？ |

**打分流程**: 接收团队状态报告 → 对照 `.Codex/SCORING-RUBRIC.md` 打分 → 任一方总分 < 7 调用 `evolution-engine`
