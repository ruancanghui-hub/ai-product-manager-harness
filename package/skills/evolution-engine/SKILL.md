# 自进化引擎（Evolution Engine）

## 目的

让角色 Skill 系统**越用越好**。本 Skill 实现了「下游验收上游 → 打分 → 不及格 → 自动改 SKILL → 下次重打分」的完整进化闭环。

**核心原则**: 每一次「上游交付物不达标」都不是浪费，而是让系统进化一次的燃料。

## 触发条件

以下场景调用本 Skill：
- 下游角色接收上游交付物时，发现质量不达标
- 用户或任何角色说「这个 PRD 写得太差了」「设计稿缺了很多状态」「接口文档对不上」
- 定期质量审计（每 3 个版本或用户要求）
- 同一个上游 Skill 被连续两次打分不及格
- 用户说「看看我们哪里最需要改进」

## 打分体系

### 评分维度（4 个维度，各 1-10 分）

| 维度 | 含义 | 1 分 | 10 分 |
|------|------|------|-------|
| **完整性** | 该有的都有了吗？ | 缺核心章节/状态/接口 | 所有要求的内容都齐了 |
| **清晰度** | 没有歧义吗？ | 多处模糊，下游要猜 | 每个点都精确，无歧义 |
| **及时性** | 按时交付了吗？ | 延期超过 3 天 | 提前或准时交付 |
| **可用性** | 下游能直接用吗？ | 需要大量返工和追问 | 拿来即用，零阻塞 |

### 综合评分

- **总分 = (完整性 + 清晰度 + 及时性 + 可用性) / 4**
- **≥ 8 分**: ✅ 优秀，直接通过
- **7.0-7.9 分**: ⚠️ 及格，记录改进建议但不阻塞
- **< 7 分**: ❌ 不及格，触发进化流程

### 谁给谁打分（上下游评分关系）

| 打分方（下游） | 被打分方（上游） | 评什么 |
|---------------|-----------------|--------|
| **design-studio** | pm-center | PRD 质量——需求是否清晰、流程是否完整、异常状态是否覆盖 |
| **dev-factory** | pm-center | PRD 技术可行性——验收标准是否可测、数据模型是否明确 |
| **dev-factory** | design-studio | 设计交付质量——状态是否全覆盖、标注是否清晰、组件是否一致 |
| **qa-gate** | dev-factory | 提测质量——自检是否诚实、构建包是否可用、已知问题是否列出 |
| **qa-gate** | pm-center | 验收标准质量——是否可测试、是否覆盖边界 |
| **ops-growth** | pm-center | 功能说明质量——是否足够运营理解功能 |
| **ops-growth** | dev-factory | 配置文档质量——后台/开关/灰度是否写清楚 |
| **chief-orchestrator** | 所有团队 | 状态报告质量——进度是否准确、风险是否不隐瞒 |
| **mgmt-office** | 所有团队 | 信息透明度——是否该上报的都上报了 |

## 进化流程（4 步闭环）

### Step 1: 打分触发
1. 下游角色接收上游交付物
2. 按 4 维度打分（1-10）
3. 如果总分 < 7 → 进入 Step 2
4. 如果总分 ≥ 7 → 记录分数到 `.ai-pm/feedback/SCORE-LOG.md`，流程结束

### Step 2: 问题诊断
1. 分析低分维度，定位具体问题：
   - 「完整性 4 分」→ 缺了什么？哪个章节/状态/接口没写？
   - 「清晰度 3 分」→ 哪里模糊了？下游在哪个点产生了歧义？
   - 「可用性 5 分」→ 下游拿到后做了多少额外工作才用起来？
2. 写诊断报告，格式：

```markdown
## 诊断: [被打分Skill] ← [打分Skill] — YYYY-MM-DD

**评分**: 完整性 X / 清晰度 X / 及时性 X / 可用性 X = 总分 X.X ❌

**具体问题**:
1. [问题1]: 具体描述，附证据（如 PRD 第 X 节缺少错误状态描述）
2. [问题2]: ...

**根因**: 是上游 Skill 的工作流程缺失，还是执行不到位？

**建议改进**: 上游 Skill 的哪个章节/流程需要怎么改？
```

3. 写入 `.ai-pm/feedback/YYYY-MM-DD-<被打分skill>-score.md`

### Step 3: Skill 改进（关键步骤）
1. 读取被打分 Skill 的 `SKILL.md`
2. 根据诊断定位到具体章节：
   - 如果是**工作流程缺失**（比如 pm-center 的 PRD 流程没要求写异常状态）→ 在该流程步骤中添加检查项
   - 如果是**验收标准缺失**（比如 SKILL 的验收标准没包含这个点）→ 在验收标准 checklist 中补上
   - 如果是**禁止事项不够强**（比如没有禁止「不写异常状态就交付」）→ 在禁止事项中增加
   - 如果是**硬停止条件不足**（比如应该在 XX 情况下拒绝交付但没有）→ 增加硬停止规则
3. 修改 `SKILL.md`，遵循以下格式：
   - 新增内容前加 `> 🔄 进化于 YYYY-MM-DD: [下游反馈] [问题摘要]`
   - 保留原有内容，追加改进项（不删除原有核心逻辑）
4. 在被打分 Skill 的变更记录区写入：
```markdown
## 🔄 进化记录
| 日期 | 触发方 | 问题 | 改进内容 |
|------|--------|------|----------|
| YYYY-MM-DD | design-studio | PRD缺少异常状态描述 | 在Phase C流程中新增「必须覆盖异常分支」检查项 |
```

### Step 4: 重打分验证
1. 改进后的下一次交付，下游**必须重新打分**
2. 对比改进前后分数：
   - 分数提升 → ✅ 进化有效，记录到 `.ai-pm/feedback/EVOLUTION-LOG.md`
   - 分数不变或下降 → ❌ 进化无效，回退修改，重新诊断

## 进化记录索引

### `.ai-pm/feedback/FEEDBACK-INDEX.md`
每个打分事件生成一条索引：
```markdown
| 日期 | 打分方 | 被打分方 | 总分 | 状态 | 链接 |
|------|--------|----------|:----:|------|------|
| 2026-07-01 | design-studio | pm-center | 5.5 | 已进化 | [详情](2026-07-01-pm-center-score.md) |
| 2026-07-05 | dev-factory | design-studio | 8.0 | 通过 | - |
```

### `.ai-pm/feedback/EVOLUTION-LOG.md`
每次有效进化记录一条：
```markdown
| 日期 | 被改进Skill | 改进前均分 | 改进后均分 | 改进内容摘要 |
|------|------------|:----------:|:----------:|-------------|
| 2026-07-02 | pm-center | 5.5 | 8.2 | PRD流程增加异常状态必填检查 |
```

## 周期审计

每 3 个版本或用户说「做质量审计」时：

1. 扫描 `.ai-pm/feedback/SCORE-LOG.md`
2. 找出：
   - **最低分 Skill** → 重点改进
   - **分数下降趋势的 Skill** → 预警
   - **连续 3 次低分的 Skill** → 重大重构建议
3. 输出审计报告：
```markdown
## 质量审计报告 — YYYY-MM-DD
- 平均分最高 Skill: xxx (8.5)
- 平均分最低 Skill: xxx (5.2) ⚠️ 需重点改进
- 分数趋势: pm-center 连续3版下降 (8.0→7.2→6.5)
- 改进建议: ...
```

## 与其他 Skill 的协作

| 场景 | 动作 |
|------|------|
| design-studio 收到 PRD，发现缺了 3 个异常状态 | design-studio 打分(完整性=4)→不及格→evolution-engine 触发→改 pm-center/SKILL.md→重打分 |
| dev-factory 收到设计稿，发现没标注深色模式 | dev-factory 打分(完整性=5)→不及格→evolution-engine 触发→改 design-studio/SKILL.md |
| qa-gate 提测时发现自检报告作假 | qa-gate 打分(可用性=2)→不及格→evolution-engine 触发→改 dev-factory/SKILL.md 自检流程 |

## 输出要求

本 Skill 执行后必须产出：
1. **评分记录** — `.ai-pm/feedback/SCORE-LOG.md` 更新
2. **诊断报告** — `.ai-pm/feedback/YYYY-MM-DD-<skill>-score.md`（如不及格）
3. **Skill 改进** — 对应 `SKILL.md` 已更新（如不及格）
4. **进化记录** — `.ai-pm/feedback/EVOLUTION-LOG.md` 更新（改进有效后）

## 验收标准

- [ ] 下游打分有明确的证据支撑（不是「感觉不好」，而是「第 X 节缺少 Y」）
- [ ] 不及格触发了 SKILL.md 实质性改进（不是改了个措辞应付）
- [ ] 改进后重打分，分数有提升
- [ ] 进化记录可追溯——知道哪个版本改了什么、因为什么
- [ ] 不会因为一次低分就过度修改（保持 Skill 的核心逻辑稳定）

## 禁止事项

- ❌ 打分不附证据（「就是不行」——不行在哪里？）
- ❌ 改 SKILL 只改措辞不做实质改进
- ❌ 分数虚高（「给个面子打 8 分吧」——下游放水 = 系统退化）
- ❌ 只打分不追踪（打完分了没记录，下次同样问题继续出现）
- ❌ 一次低分就大幅重写 Skill（进化是小步快跑，不是推倒重来）
