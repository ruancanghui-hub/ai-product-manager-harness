# 总调度（版本经理/大PMO/项目集经理）— 日常工作总结

> 不是做具体执行的，而是把所有人拧成一股绳的那个人。
> APP 从 0 到 1 再到 N，每个环节的交接、节奏、冲突、风险——由总调度兜底。

---

## 一句话定位

**让 30+ 个角色不是各干各的，而是像一条流水线一样紧密咬合。**

总调度不问「这个按钮什么颜色」「这个接口延迟多少毫秒」，但他/她必须知道：
- 设计稿什么时候出？会不会卡住开发？
- 后端接口和客户端开发谁等谁？
- 测试手里还有多少 bug，发布时间要不要延期？
- 运营活动物料到位没有？别等上线了才发现没素材。

## 总调度的核心工作模型：5 个「盯」

### 1. 盯节奏 —— 版本迭代的「心跳」不乱

APP 开发一旦进入多团队并行，最大的敌人不是技术难题，而是**节奏错位**。

**总调度要确保的节奏：**

```
迭代 Day 1-2：需求宣讲（PM 把 PRD 讲给所有人听）
迭代 Day 3-4：设计评审（UX/UI 出稿，开发确认可实现性）
迭代 Day 5-6：技术方案评审（架构师 + 开发输出技术方案）
迭代 Day 7-12：开发并行冲刺（客户端 + 后端 + 算法同步写）
迭代 Day 13：提测日（开发代码冻结，包给测试）
迭代 Day 14-18：测试周期（功能→兼容→性能→安全）
迭代 Day 19：灰度发布（1%→5%→观察数据）
迭代 Day 20-21：全量上线 + 线上监控 + 运营活动同步上线
```

**总调度每天问自己的三个问题：**
1. 今天该推进的环节，推进了吗？（设计稿今天该出，出了没？）
2. 明天该启动的环节，前置条件齐了吗？（明天开发开始写，接口文档好了没？）
3. 哪个环节现在卡住了，会污染下游吗？（设计延期 2 天，开发要不要先做别的？）

### 2. 盯依赖 —— 别让「我在等 XX」成为口头禅

这是总调度最核心的价值。每个角色的产出，都可能是另一个角色的输入。

**全链路依赖图（总调度脑海中的版图）：**

```
PM 的 PRD ──→ UX 的交互稿 ──→ UI 的视觉稿 ──→ 客户端开发
                         └──→ 后端开发（接口设计）
                                       └──→ 客户端联调
                                              └──→ 功能测试
                                                     └──→ 兼容性测试
                                                            └──→ 合规测试
                                                                   └──→ 灰度
                                                                          └──→ 运营活动上线
                                                                                 └──→ 数据回收
分析师的埋点需求 ──→ 客户端埋点开发 ──→ 测试验证埋点 ──→ 分析师 ──→ PM 下一迭代
                                                                         ^^^^^^^^^^^^^^^
算法模型 ──→ 后端集成 ──→ 客户端接入 ──→ 算法效果验证 ──→ 推荐上线
法务审核用户协议 ──→ 合规测试 ──→ 应用商店提交 ──→ 审核通过 ──→ 全量
```

**总调度的关键动作：**
- 迭代启动前，画出本迭代的依赖链路图，标出所有「交付物 A → 接收方 B」的箭头
- 每个箭头标注预期交付日期
- 每个交付日前 2 天，私聊确认：「XX，你周三要给 YY 的那个东西，能按时吗？」
- 一旦某个节点出现延期风险，立刻评估下游影响，做三件事之一：**帮上游加速 / 调整下游排期 / 做预案（如先给 Mock 数据）**

### 3. 盯瓶颈 —— 找到整个链条中最慢的那一环

任何一个环节慢了，整个版本就慢了。总调度要时刻识别瓶颈。

**常见瓶颈类型及总调度应对策略：**

| 瓶颈 | 表现 | 总调度动作 |
|------|------|------------|
| **设计瓶颈** | UX/UI 一个人跟 3 条线，稿子出不来 | 协调设计负责人做优先级排序：哪条线先出，哪条线可以简化为复用组件 |
| **客户端瓶颈** | iOS 开发 2 人，需求 5 个并行，写不完 | 和 PM 确认 P0/P1/P2 优先级，P2 挪到下个迭代；协调跨端开发支援 |
| **后端瓶颈** | 后端接口延期，客户端等接口没法联调 | 推动后端先出接口文档 + Mock 数据，客户端先写 UI 层和假数据，接口就绪后再联调 |
| **测试瓶颈** | 提测包 bug 太多打回，测试时间被压缩 | 推动开发自测质量提升（提测前必须过自测 Checklist）；协调增加测试人力；高风险模块单独提测 |
| **审核瓶颈** | App Store 审核周期不确定，版本被迫延期 | 提前 1-2 周提交审核；审核期并行准备下一个迭代 |
| **法务瓶颈** | 用户协议/隐私政策审核周期长 | 提前 2 周把文案提交法务，不等最后关头 |

### 4. 盯信息 —— 让该知道的人知道

大厂 APP 团队几十上百人，最大的效率损耗不是技术，而是**信息不对称**。

**总调度的信息分发工作：**

| 频率 | 动作 | 内容 |
|------|------|------|
| **每日** | 站会纪要 | 发到全员群：今天关键进展、阻塞项、需要谁配合 |
| **每周** | 周报 | 本周进度 vs 计划、风险项（红黄绿标记）、下周关键里程碑 |
| **每个里程碑** | 里程碑检查邮件 | 需求冻结确认 / 代码冻结确认 / 提测确认 / 灰度确认 |
| **突发** | 即时同步 | 某个关键环节延期→立刻通知全部下游；线上故障→拉应急群；审核被拒→同步 PM + 法务 + 开发 |
| **每版本** | 版本复盘 | 这个版本延期了吗、为什么、哪个环节最卡、下次怎么改进 |

**关键原则：对下游，提前告知坏消息（"后端会晚2天"），好过最后突然通知。**

### 5. 盯质量 —— 每个节点不放水

总调度不是质量负责人（那是测试和 TL 的职责），但总调度**守住质量门**。

**每个阶段的质量门：**

```
需求评审 ──→ 出口标准：PRD 写清楚了，开发没有「需求不清楚」的疑问
    ↓
设计评审 ──→ 出口标准：交互稿开发理解一致，异常状态覆盖完整
    ↓
技术评审 ──→ 出口标准：接口定义、数据库变更、风险点、回滚方案都评审通过
    ↓
开发完成 ──→ 出口标准：自测 Checklist 全部通过，P0 用例自测无阻塞 bug
    ↓
提测     ──→ 出口标准：打包成功，冒烟通过，否则打回开发
    ↓
测试完成 ──→ 出口标准：P0/P1 bug 清零，P2 bug 有明确处理计划，性能/兼容/合规通过
    ↓
灰度上线 ──→ 出口标准：灰度期间无 P0 告警，核心指标无异常下跌
    ↓
全量上线 ──→ 出口标准：全量后 24h 内核心指标稳定，无新增严重崩溃
```

**总调度的质量把控姿态：**
- 不是每个环节亲自检查，而是**确保每个环节的负责人自己检查了**
- 在质量门处问一句：「XX，你这个环节的出口标准都过了吗？」
- 如果没过强行往前冲（"来不及了先提测吧"），总调度需要站出来说不——并解释推迟的代价 vs 带着 bug 上线的代价

## 总调度一周的真实日常

### 周一：对齐周

| 时间 | 事项 |
|------|------|
| 09:30 | 看上周数据报表（崩溃率/DAU/留存），有没有线上问题需要本周紧急修复 |
| 10:00 | 站会——30+ 人的大站会走一圈，每人 30 秒说三句话（昨天/今天/阻塞） |
| 10:30 | 会后处理站会暴露的阻塞项：谁等谁，立刻拉群解决 |
| 11:00 | 和 PM 1-on-1：这周需求有没有变更？下个迭代的需求优先级排好了吗？ |
| 14:00 | 周会——拉各端负责人（客户端 TL/后端 TL/测试负责人/设计负责人/PM）过本周里程碑 |
| 16:00 | 检查依赖链路图：本周要交付的关键产出物，负责人确认能按时吗？ |
| 17:00 | 输出周报：本周目标 vs 当前进度，红黄绿风险标记，发全员 |

### 周二-周四：执行推进

| 时间 | 事项 |
|------|------|
| 09:30 | 看各端 Jira/TAPD 看板，有没有任务卡在某个状态超过 2 天 |
| 10:00 | 站会 |
| 10:30-12:00 | 深度跟进关键路径任务（比如：支付模块是本次版本最难的部分，重点盯） |
| 14:00 | 参加技术评审/设计评审（总调度不决策，但必须知道结论和对排期的影响） |
| 16:00 | 检查测试进度：bug 数量趋势是收敛还是发散？新增 bug 速率在下降吗？ |
| 17:30 | 和任何一个出现延期风险的环节负责人 1-on-1，了解真实情况和需要什么帮助 |

### 周五：收口+规划

| 时间 | 事项 |
|------|------|
| 09:30 | 站会（周五站会短，15 分钟过完） |
| 10:00 | 本周进度最终确认：该交付的都交付了吗？没交付的什么原因，下周一能补吗？ |
| 14:00 | 下周一发布/灰度的准备工作检查（checklist 逐条过） |
| 16:00 | 和 TL 们碰头：下周资源有没有变动（请假/借调/新需求插入） |
| 17:00 | 本周收尾：更新项目看板、整理下周关键日程、周五下班前让所有人知道下周一早上要干嘛 |

### 发布日（不固定，迭代最后几天）

| 时间 | 事项 |
|------|------|
| 08:00 | 提前到岗，确认发布包就绪、灰度配置就绪 |
| 09:00 | 发布前 checklist 最后一轮确认（各端负责人签字） |
| 10:00 | 开始灰度（1%用户） |
| 10:30 | 盯监控大盘：崩溃率、接口成功率、核心业务指标 |
| 11:00 | 灰度无异常 → 放量到 5% → 继续观察 |
| 14:00 | 灰度无异常 → 放量到 50% |
| 16:00 | 全量确认 → 通知运营同步上线活动 → 通知客服团队注意用户反馈 |
| 18:00 | 下班前再看一次监控 → 准备回滚方案（有备无患） |

## 总调度和各个角色的协作模式

| 对接角色 | 协作频率 | 协作内容 |
|----------|:--------:|----------|
| **PM** | 每天 | 对齐需求变更、优先级调整、下版本规划 |
| **交互/视觉设计师** | 每 2-3 天 | 确认设计交付时间、设计评审时间 |
| **iOS/Android TL** | 每天 | 开发进度、阻塞项、人员调配 |
| **后端 TL** | 每天 | 接口交付进度、和客户端的联调计划 |
| **测试负责人** | 每天 | bug 趋势、测试进度、提测/打回决策 |
| **架构师** | 每迭代 | 技术评审时间安排、重大技术风险 |
| **算法团队** | 每迭代 | 模型交付时间和集成计划 |
| **DevOps** | 每版本 | 打包时间、灰度策略、发布窗口 |
| **法务** | 每版本 | 协议审核进度、隐私合规确认 |
| **运营** | 发布前 | 活动物料准备、活动上线时间对齐、渠道投放同步 |
| **数据分析师** | 发布后 | 版本数据回收时间、复盘报告时间 |
| **客服** | 发布后 | 用户反馈收集、重大问题上报机制 |
| **总监/业务负责人** | 每周/每版本 | 进度汇报、风险升级、资源申请 |

## 总调度的核心能力清单

| 能力 | 为什么重要 |
|------|------------|
| **全局视野** | 看得见整条链路，不被局部噪音带偏 |
| **风险嗅觉** | 在问题还没发生时闻到味道——某人回复慢了、bug曲线不收敛、有人开始说「应该能赶上」 |
| **沟通效率** | 能把技术、产品、设计、运营的不同语言翻译成对方能听懂的版本 |
| **推动力** | 不是自己有权力命令谁，而是能说服、协调、借力推动事情往前走 |
| **冷静** | 线上炸了、审核被拒了、核心开发离职了——总调度不能慌 |
| **责任心** | 总调度是唯一一个「没有直接产出、但所有产出都和你有关系」的角色，出了问题没有「不关我事」 |

## 附：总调度常用的「灵魂三问」

在关键节点上，总调度会反复问这三个问题：

1. **「这个交付物，下一环的人收到了吗、看懂了吗、能用吗？」**
   — 不是交了就完了，要确认下游接收成功

2. **「如果 XX 延期 2 天，对最终发布时间的影响是什么？」**
   — 提前推演，不是事后救火

3. **「现在整个链条中，最可能炸的雷是哪个？」**
   — 找到最脆弱的那一环，提前加固

## 大厂差异

| 维度 | 阿里 | 腾讯 | 谷歌 | 苹果 |
|------|------|------|------|------|
| **岗位名称** | 大PMO/版本经理 | 项目经理/交付PM | TPM | EPM |
| **人数** | 一个 APP 1-3 个 PMO | 1-2 个 PM | 每个产品线 1 个 TPM | 每个项目 1-2 个 EPM |
| **权力** | 协调权，非人事权 | 协调权，TPM 文化不强 | 技术话语权强，能影响技术决策 | 权力极大，能决定发布时间和资源分配 |
| **工具** | 钉钉+Aone+TAPD | 企业微信+TAPD | 内部工具+Google Docs | 内部保密工具 |
| **特色** | 强流程驱动，阿里的 PMO 体系成熟 | 微信侧 PM 文化弱，游戏侧交付 PM 强势 | TPM 偏技术，能参与架构讨论 | EPM 是「推动一切发生」的核心角色，保密文化下信息管控极严 |
