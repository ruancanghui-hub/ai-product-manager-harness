# 一、产品中心 — 日常工作总结

> 定方向：做什么、解决什么问题、为什么用户买单

---

## 1. 产品负责人 / 主 PM

**一句话定位**：APP 的「总导演」，对产品最终成败负责。

### 📋 日常核心工作

| 时间段 | 工作内容 |
|--------|----------|
| **每日** | 看数据报表（DAU/留存/转化/崩溃率），异常波动立即溯源；回复各端（设计/开发/测试/运营）的即时消息和阻塞问题 |
| **每周** | 主持需求评审会，向开发+设计讲清楚本期需求背景、目标、逻辑；参加迭代站会，跟进开发进度，现场拍板需求疑问；过竞品更新，输出竞品分析简报 |
| **每迭代（2-4周）** | 输出/更新 PRD（产品需求文档），包含：业务背景、用户场景、功能流程图、交互逻辑、异常分支、数据埋点要求、验收标准；排优先级（P0/P1/P2），对齐开发资源 |
| **每版本** | 版本上线后盯数据（核心指标+业务指标），判断功能好坏；收集用户反馈（App Store 评论、客服工单、社群），提炼下版本需求 |
| **每季度** | 制定产品路线图（Roadmap），对齐业务目标（营收/日活/留存/GMV）；向业务负责人汇报版本成果和下阶段规划 |

### 🔧 常用工具
- 原型：Figma / Sketch / Axure
- 文档：飞书文档 / Notion / Confluence / Google Docs
- 项目管理：Jira / TAPD / Linear / Asana
- 数据：神策 / GrowingIO / Google Analytics / 自建 BI 平台

### 🏢 大厂差异
- **阿里/腾讯**：PM 分得更细（用户端 PM、商家端 PM、中台 PM），一个功能可能 3 个 PM 各管一层
- **谷歌/苹果**：PM（Google 叫 Product Manager，苹果叫 Product Marketing Manager）更偏战略层，工程团队自主权更大，PM 不写详细 PRD 而是写 PRD（Product Requirement Document）精简版 + OKR 驱动

---

## 2. 后台/数据 PM

**一句话定位**：让内部人员能「看清数据、管好后台」。

### 📋 日常核心工作

| 工作项 | 具体内容 |
|--------|----------|
| **埋点设计** | 定义页面曝光、按钮点击、业务流程各节点的埋点事件和参数；维护埋点文档，跟进开发埋点实现，校验数据上报准确性 |
| **数据看板** | 搭建业务看板（实时大屏、日报、周报），给运营/产品/管理层提供可视化数据；对接数据分析师需求，确保口径统一 |
| **后台管理系统** | 设计运营后台（内容管理、用户管理、活动配置、推送后台）、客服后台、审核后台的功能需求；输出后台 PRD，对接后台开发 |
| **权限与报表** | 设计 RBAC 权限体系；定制各类业务报表（订单报表、用户增长报表、收入报表） |
| **日常沟通** | 对接运营团队收集后台使用痛点；对接数据团队对齐统计口径；对接后端开发评审后台接口方案 |

### 🏢 大厂差异
- **阿里**：数据 PM 和后台 PM 常分开，后台 PM 专注卖家/商家后台（千牛等）
- **中小公司**：这个角色通常合并到主 PM 或后端负责人身上

---

## 3. 商业化 PM

**一句话定位**：让 APP 赚钱——广告、会员、付费、电商变现的设计者。

### 📋 日常核心工作

| 工作项 | 具体内容 |
|--------|----------|
| **广告系统** | 设计广告位（开屏、信息流、Banner、激励视频）、广告频次控制、竞价/定价策略、广告样式规范 |
| **会员/订阅** | 设计会员权益体系、定价策略、免费试用→付费转化路径、续费提醒机制 |
| **付费功能** | 设计虚拟商品（表情包、皮肤、礼物）、增值服务（云存储、高级滤镜）、付费门槛和定价 |
| **电商变现** | 设计直播带货流程、商品橱窗、佣金分成规则、优惠券/满减营销工具 |
| **数据驱动** | 监控 ARPU/ARPPU、付费率、广告填充率/eCPM、会员续费率；AB 测试不同定价/权益方案 |
| **合规** | 确保支付合规（苹果 IAP 30% 抽成规则、安卓支付牌照）、广告合规（不诱导点击、隐私合规） |

### 🔧 常用工具
- 广告平台：广点通/巨量引擎 / Google AdMob / Facebook Audience Network
- 支付：Apple IAP / Google Play Billing / 支付宝/微信支付

### 🏢 大厂差异
- **腾讯/阿里**：商业化 PM 是一个独立大团队（几十人），专人负责广告策略、会员运营、电商活动
- **谷歌/苹果**：更偏平台生态抽成（App Store/Google Play 的 15%-30%），自有 APP 商业化相对克制（苹果更重硬件+服务收入）

---

## 4. 海外 PM

**一句话定位**：让 APP 在全球不同市场都能「落地生根」。

### 📋 日常核心工作

| 工作项 | 具体内容 |
|--------|----------|
| **本地化适配** | 多语言翻译管理（不是直译，要符合当地文化）；RTL 布局适配（阿拉伯语）、货币/日期/单位格式适配 |
| **海外合规** | GDPR（欧盟隐私法）、CCPA（加州隐私法）、COPPA（儿童保护）、DSA（欧盟数字服务法）；各国数据跨境传输要求 |
| **渠道适配** | Google Play / App Store 海外商店页面优化（标题、关键词、截图、描述）；对接海外应用商店审核规范 |
| **市场调研** | 目标国家用户习惯、竞品分析、文化禁忌；海外用户增长策略（地推、KOL、社媒） |
| **日常协作** | 对接海外运营团队（或当地办公室）；同步国内版本功能，判断哪些需要本地化改造 |
| **法务对接** | 和法务团队一起审核用户协议、隐私政策的各国版本 |

### 🏢 大厂差异
- **阿里（Lazada/AliExpress）/腾讯（PUBG Mobile/王者荣耀海外）**：海外 PM 团队常驻目标国家，不做远程翻译
- **谷歌/苹果**：产品天生全球化，PM 团队本身就分布在全球，本地化是默认能力而非独立岗位

---

## 5. 用户研究员

**一句话定位**：用科学方法验证「用户到底要什么」。

### 📋 日常核心工作

| 工作项 | 具体内容 |
|--------|----------|
| **用户访谈** | 招募目标用户，进行一对一深度访谈（线上或线下）；记录用户痛点、使用场景、对现有功能的评价 |
| **问卷调查** | 设计问卷（定量研究），通过 APP 内推送或邮件触达；回收≥1000 份有效样本，统计显著性分析 |
| **可用性测试** | 邀请用户操作产品原型，观察操作路径、卡点、困惑点；记录任务完成率、操作时间、错误次数，输出可用性问题清单 |
| **用户画像** | 基于访谈+数据，输出 Persona（用户画像卡片）：年龄、职业、场景、核心需求、痛点、行为特征 |
| **竞品用户分析** | 分析竞品用户评价（App Store/Google Play 评论），提炼用户对竞品的满意点和不满点 |
| **输出报告** | 每项研究输出研究报告：研究目的→方法→样本→发现→建议→给 PM/设计的行动项 |
| **日常协作** | 和 PM 对齐研究需求（什么时候研究什么）；和设计师协作完成可用性测试；和数据分析师交叉验证定性+定量结论 |

### 🔧 常用工具
- 问卷：问卷星 / Typeform / Google Forms
- 用户测试：Lookback / UserTesting / Maze
- 分析：SPSS / Excel / 神策用户画像模块

### 🏢 大厂差异
- **阿里/腾讯**：有独立 UED（用户体验设计）部门，用研直属 UED，不归 PM 管，保持中立
- **谷歌**：UX Researcher 标配，且级别很高（L5-L7），直接嵌入产品团队，研究方法论非常严谨（大样本 AB + 定性深访结合）
- **苹果**：用研团队高度保密，外人很少知道具体运作，但产品体验打磨之极致反映其投入之深
- **中小公司**：无此岗位，PM 兼做

## 6. 数据分析师 DA

**一句话定位**：用数据回答「功能做得好不好、接下来该做什么」。

### 📋 日常核心工作

| 工作项 | 具体内容 |
|--------|----------|
| **搭建指标体系** | 定义北极星指标（North Star Metric）、一级指标（DAU/留存/收入）、二级指标（功能渗透率/点击率/转化率）；维护指标字典，确保全公司口径统一 |
| **日常监控** | 每天早上看核心指标看板，发现异常波动（涨跌超过阈值）；异常分析：是技术问题（崩溃/接口超时）、运营问题（活动结束）、还是产品问题（新功能劝退用户） |
| **版本分析** | 版本上线后，对比新老版本核心指标（AB 实验或前后对比）；新功能渗透率、使用频次、留存影响分析；输出版本复盘报告 |
| **AB 实验** | 设计实验方案（实验组/对照组、样本量计算、观察指标、实验时长）；实验上线后每日监控，实验结束输出显著性检验结论和建议 |
| **专项分析** | PM 提需求："为什么最近注册转化率掉了？" → 分析师漏斗分析定位流失环节 → 分用户群对比 → 提出假设 → 数据验证 |
| **取数支持** | 给 PM/运营提 SQL 取数需求（大厂有专门取数工具或数据平台，自助查）；复杂分析需求排期做 |
| **数据产品** | 和后台 PM 一起优化数据看板，让业务方自助看数；搭建自动预警规则，指标异常自动告警 |

### 🔧 常用工具
- SQL：MySQL / Hive / Presto / Spark SQL
- BI：Tableau / FineBI / 百度 Sugar / 自建 BI 平台
- 分析：Python（Pandas/Jupyter）/ R / Excel
- 埋点：神策 / GrowingIO / Amplitude / Mixpanel

### 🏢 大厂差异
- **阿里**：数据分析师分两类——业务分析（贴近业务）和技术分析（偏数据开发/算法），后者更像数据工程师
- **腾讯**：数据团队独立，分析师按 BG 分配，微信/QQ/游戏各有专门的 DA 团队
- **谷歌**：Data Scientist 或 Product Analyst，重视统计学严谨性，AB 实验文化极强，几乎所有改动都上实验
- **中小公司**：PM 自己写 SQL 看数，没有专职 DA
